## Subject-centric intelligence — implemented (ADR-003)

The decision is recorded in [`ADR-003`](./adr/ADR-003-subject-centric-intelligence.md); the sequence below is now implemented, verified against source for this update. Bounded to this repository — no BrandOS-side change was required:

- **Done.** `subject_type` discriminator added alongside the existing nullable `user_id`/`workspace_id` columns on `learnings`, plus (newly, since these previously had no workspace concept at all) `hypotheses`, `signals`, and `profiles` — `migrations/004_subject_centric_intelligence.sql`, following `002_workspace_learning_owner.sql`'s pattern.
- **Done.** The Learning Pipeline's classes (`SignalExtractor`, `ObservationBuilder`, `HypothesisEngine`, `LearningValidator`, `ProfileBuilder`) now accept a `SubjectRef` (`types/subject.ts`) rather than an assumed user id — via new `...ForSubject` methods on `UserIntelligenceDomain`; the original User-scoped method names are unchanged, thin wrappers over the new ones.
- **Done.** `ObservationInput → Signal[]` translation built (`SignalExtractor.extractFromObservation()`) — workspace-scoped observations now enter the pipeline at `SignalExtractor`, the same way a `FeedbackEvent` does, instead of being written directly as an already-classified `Learning` row. `context/observationToWorkspaceLearning.ts` (the module this superseded) has been removed. `FeedbackProcessor.processObservation()` is the new Workspace-subject orchestrator; `CognitionProviderImpl.observe()` calls it.
- **Done.** Workspace-subject identity-synthesis path added to `ContextBuilder` (`context/identitySynthesis.ts`), reusing `NarrativePlanner`'s authority-ordered composition pattern (`ARCHITECTURE.md` §10) rather than inventing a second one.
- **Done.** Known Contract Gap #2 closed via a Knowledge-Ingest-style path for explicit workspace configuration (`IntelligenceOS.ingestWorkspaceConfiguration()` → `KnowledgeIntelligenceDomain.upsertWorkspaceConfiguration()`), per `ADR-003` §2.4. `ContextBuilder` applies it ahead of Learning-derived voice.

Together, this closes both of `IMPLEMENTATION_STATUS.md`'s former identity/voice Known Issues as one generalization, not two separate patches. `pnpm -r typecheck`, `pnpm --filter @intelligence-os/core test`, and `pnpm --filter @intelligence-os/core run check:boundaries` are all clean against this change — see `IMPLEMENTATION_STATUS.md` for exact counts.

**Resolved since, in the ADR-003 audit-closure session (see `ADR-003` §9, `IMPLEMENTATION_STATUS.md` §3 third session):**
- `IntelligenceOS.ingestWorkspaceConfiguration()` is now part of `IIntelligenceProvider`, and reachable at `POST /v1/workspace-configuration` (a sibling route to `CognitionProvider`, not a sixth `CognitionProvider` operation).
- `identity` synthesis now also draws on Knowledge (an explicit `identityConfiguration`, symmetric with `voiceConfiguration`), not only Learnings.

**Still not done, deliberately deferred as separate decisions:**
- **Ordinary document-extracted Knowledge still has no path into `voice`/`identity` synthesis.** Only the narrow, explicit `upsertWorkspaceConfiguration()` path does. The largest remaining piece of `ADR-003` §2.3's own text — closing it needs a real design decision (which extracted fields matter, at what confidence, ranked how against explicit configuration and Learnings), not a mechanical read-path fix. See `IMPLEMENTATION_STATUS.md` §5/§6.
- **A contributing User's identity is not composed into a Workspace's additively.** `CognitionRequest` still carries no `userId` to identify one (ADR-003 §2.3, §5) — today's Workspace identity is synthesized purely from the workspace's own Knowledge and Learnings. Revisit if/when a genuinely multi-user consumer needs it (see the `WorkspaceIntelligenceDomain` Phase 2 item below).
- **No live-database verification.** `migrations/004_subject_centric_intelligence.sql` has not been applied to any live Supabase project in this environment (none is configured) — see the near-term schema-migration item above and `IMPLEMENTATION_STATUS.md` §4.

**Do not** add an optional `userId` to `CognitionRequest` as a narrower substitute for the above — considered and explicitly rejected in `ADR-003` §5.

## Mid-term (the two open cross-platform contract questions)

Both are documented; gap 2's target direction is decided and implemented in IntelligenceOS (`ADR-003`), gap 1's is not — see `PLATFORM_CONTRACT.md` §3 and `packages/cognition-contract/README.md`:

1. **Raw-signal review UI.** BrandOS needs a way to list reviewable signals for a human approve/reject workflow; the current contract only supports acting on a signal by opaque id. Candidate direction: a narrowly-scoped, already-summarized "reviewable items" read added to the contract (classification + confidence + a pre-rendered display string — still no raw content). Still needs a deliberate decision.
2. **Explicit brand-voice/identity configuration ingestion.** Direction decided, implemented, and now reachable in IntelligenceOS: this is Knowledge, not an `observe()`-reported outcome — see the Subject-centric intelligence section above. `IIntelligenceProvider.ingestWorkspaceConfiguration()` and `POST /v1/workspace-configuration` are both live; whether BrandOS's own admin UI calls either yet is outside this repository's scope to verify.

## Mid-term (consumer-side — largely outside this repository)

The original three-epic plan's third epic (BrandOS Adoption: dual-write, shadow-mode parity validation, progressive rollout, legacy-package retirement) was never executed under that name — work instead continued directly as Milestones 2–4 (see `IMPLEMENTATION_STATUS.md` §3), which built the HTTP contract layer this document's mid-term items above assume exists. ## Longer-term (larger, activation-gated capabilities)

These are documented, designed-for capabilities waiting on their stated activation trigger — not missing features, in the sense the architecture is explicit that they shouldn't be built before their trigger condition is real:

- **`RelationshipIntelligenceDomain` activation.** Currently fully inert for every real operation (every method but one throws `DomainNotActivatedError`). Activates at ≥3 external artifacts with named recipients for a given user, or an explicit onboarding signal — see `ARCHITECTURE.md` §6 and `archive/foundations/Intelligence_Contracts.md` §J.2–J.3 for the full activation model. **The trigger-check itself is now implemented** (`ArtifactIntelligenceDomain.countArtifactsWithNamedRecipients()` / `RelationshipIntelligenceDomain.checkActivationTrigger()` — `IMPLEMENTATION_STATUS.md` §3, fifth session) — it does not itself activate anything; building real named-relationship storage and read paths so the domain's other methods stop throwing is separate, larger Phase 2 feature work, still unscheduled.
- **Visual mood → Learning promotion.** `ADR-001` §5 requirement: once visual feature extraction has run enough times for a given user, promote consistent visual patterns into proper `Learning` rows the same way textual patterns are promoted. No consumer exercises this path yet, so it's unscheduled rather than blocked.
- **`WorkspaceIntelligenceDomain` Phase 2** — multi-user governance, shared-vocabulary enforcement, a standards board, and Immutability Rule enforcement at the workspace layer, beyond today's Phase 1 single-user-workspace scope. Composing a known contributing user's identity into a workspace's (`ADR-003` §2.3) makes this more consequential than it was before the workspace subject existed — re-evaluate this item's priority, not its scope, once a genuinely multi-user consumer (a multi-editor MarketingOS workspace, a multi-reviewer ProposalOS workflow) is actually scheduled.
- **A third Subject type (e.g., an Organization rolling up several workspaces).** Explicitly not scheduled and explicitly not designed against speculatively — `ADR-003` §5 rejects introducing this now for the same reason `ARCHITECTURE.md` §2 rejects it for any other capability: build it when a real consumer demonstrates the need, following `ArchetypeType`'s open-string-union precedent, not before.
- **Workspace-scoped `AudienceProfile`.** Today `AudienceProfile` is user-scoped only. No current consumer need demonstrates a workspace-level equivalent; not scheduled.
- **The remaining six `CognitionContext` target sections** (`knowledge`, `reasoning`, `positioning`, `audience`, `narrative`, `guidance`) — add only when a real BrandOS-side consumer need drives a specific field, per the evolution rules in `PLATFORM_CONTRACT.md` §5. Do not pre-build these speculatively.
- **Repository context generation tooling.** `archive/planning/Repository_Context_Strategy_Proposal.md`'s `.context/*.generated.md` design (mechanically-generated, always-current file inventories, boundary-violation audits, and test-coverage maps) was never built, but several Known Issues this consolidation found by hand — stale docblocks, an undetected `ADR-002` migration gap, a misplaced `AGENT_CONTEXT.md` — are exactly the class of drift that design was meant to catch automatically. Worth revisiting if the repository grows past the point where hand-maintained documentation reliably stays accurate on its own.
