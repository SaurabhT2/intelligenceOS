# IntelligenceOS — Repository Context Strategy

This document defines what generated context artifacts IntelligenceOS maintains, why each one exists, who/what consumes it, and how it stays current. It is the companion to `INTELLIGENCEOS_BOOTSTRAP.md`: the bootstrap explains *how the system works and why*; the artifacts below answer *what exists right now, exactly, derived mechanically from source*.

## Design principle

`INTELLIGENCEOS_BOOTSTRAP.md` and every `AGENT_CONTEXT.md` are **hand-maintained** — they encode judgment (why a boundary exists, what a stub means, what the activation trigger for a deferred capability is). The artifacts in this document are **generated** — they encode facts mechanically derivable from the source tree (what exists, what imports what, what's covered by tests). Hand-maintained documents should never duplicate what a generator can produce faithfully; generated documents should never contain judgment a human or agent might disagree with. When the two genuinely overlap (e.g., the bootstrap's package table vs. the generated package inventory), the hand-maintained document states the *rule*, and the generated artifact states the *current instance count* — never the same fact derived twice from two different sources of truth.

All generated artifacts:
- Live under `.context/` at the repository root.
- Carry a standard header: `<!-- GENERATED — do not hand-edit. Run `pnpm context:generate` to refresh. Source commit: <sha> -->` (or the JSON equivalent as a `"_generated"` key).
- Are committed to version control (so a fresh clone is immediately self-describing without a build step) but are never hand-edited — a hand edit will be silently overwritten on the next generation run and is also a signal that the generator itself needs a fix.
- Are regenerated by a single `pnpm context:generate` script at the workspace root, runnable in CI as a "is this stale?" check (diff against a fresh run; fail if non-empty) and locally as a normal maintenance step after any change to package structure, public exports, domain table ownership, or test coverage.

## The five artifacts

### 1. `repository_context.generated.md`

**What it is:** A single-page, always-current snapshot of the whole repository — package list with one-line purpose, file counts per major directory, current public API surface (the exact exports from each package's `index.ts`), current dependency versions, and current test/coverage summary numbers. This is the file an agent should read first, before the bootstrap, when it needs to know *today's exact state* rather than *the durable rules*.

**Why IntelligenceOS specifically needs this, not a generic version:** this repository's public API surface is supposed to be small and stable (the bootstrap's Architectural Rule 7 — four methods plus a bus accessor). A generic "list every export" tool would bury that signal in noise from internal types. This artifact should specifically foreground "the four methods + their current signatures" at the top, then list everything else below — because a regression in that surface (a fifth public method appearing, a signature changing) is the single highest-value thing this file can catch for IntelligenceOS.

**Generation method:** static analysis of `package.json` `exports` fields + TypeScript AST parse of each package's `index.ts` to extract the literal export list and signatures. No LLM involvement needed or wanted — this should be byte-for-byte deterministic.

**Consumed by:** any agent or engineer's first read in a session that doesn't already have repository state cached; CI, as a "does the public surface match what we expect" gate.

### 2. `package_inventory.generated.md`

**What it is:** Per-package detail one level deeper than `repository_context`: for each of the two packages, its full dependency list (internal and external), its `package.json` scripts, its file tree with one-line-per-file purpose extracted from each file's top docblock, and a real/stub status rollup per file (derived from scanning for the `✓`/`✗` docblock convention and for `PhaseNotImplementedError`/`DomainNotActivatedError` throw sites).

**Why IntelligenceOS specifically needs this:** the real/stub distinction is load-bearing throughout this codebase (Architectural Rule 10) — and it currently lives only in prose inside each file's docblock, which drifts. A generated rollup that mechanically counts "X real methods, Y stubbed methods, Z domains fully inert" per package turns a rule that's currently enforced by code review discipline into something checkable at a glance, and turns "did anyone update the docblock after implementing this stub" into a diffable fact (if the file's actual stub-status changes but the docblock claim doesn't, that's a flag this generator should surface, not silently average over).

**Generation method:** scan every `.ts` file for its leading docblock, the `✓`/`✗` markers within it, and cross-reference against actual method bodies (does this method's body consist of nothing but `throw new PhaseNotImplementedError(...)` or `throw new DomainNotActivatedError(...)`? If the docblock says ✓ but the body throws, or says ✗ but the body doesn't throw, flag it as a mismatch in a dedicated "Docblock/Implementation Drift" section at the top of the file). This cross-check is the single most valuable thing this artifact does — implement it before anything else in this file.

**Consumed by:** anyone deciding whether a method they're about to call is real or will throw; anyone doing a sprint-boundary review of what's been un-stubbed since the last one.

### 3. `pipeline_topology.generated.md`

**What it is:** A mechanically-extracted call graph for the three pipelines (Learning, Blueprint, Knowledge) — which class constructs which, which method calls which, in what order, with the actual file:line of each call site. Rendered as both a Mermaid flowchart per pipeline and a flat ordered list.

**Why IntelligenceOS specifically needs this, not a generic dependency graph:** this repository has exactly three pipelines, each with a single orchestrator class, and the bootstrap's read order (§9) and pipeline descriptions (§7, §8) describe them *as of this writing*. The moment someone reorders a step inside `BlueprintBuilder.build()`, adds a seventh blueprint-assembly step, or changes which class `FeedbackProcessor` constructs, the hand-written prose in the bootstrap silently goes stale — and pipeline step order is exactly the kind of fact a human reviewer is unlikely to notice drifted, because the code still "reads fine" locally even when the documented sequence no longer matches. A generic whole-repo dependency graph would be far too noisy (every file imports `errors.ts`); this artifact should be scoped specifically to the three orchestrators and what they call, in call order, which is a much smaller and much more actionable signal.

**Generation method:** AST parse of `FeedbackProcessor.process()`, `BlueprintBuilder.build()`, and `KnowledgeProcessor.process()` specifically, walking method calls on injected dependencies in source order, emitting a Mermaid sequence/flowchart block. This is the one artifact in this list where a deterministic AST walk is harder than the others and a lightweight LLM-assisted pass (summarizing the call graph from the parsed AST, not inventing one from scratch) is a reasonable implementation shortcut — but the *output* should still be checked against the AST mechanically (does every class named in the Mermaid diagram actually get constructed in the relevant orchestrator's constructor? does every arrow correspond to a real call site?) before being committed, so a hallucinated call never ships silently.

**Consumed by:** anyone modifying step order or adding a stage to one of the three pipelines, before they touch the bootstrap's hand-written description of that pipeline; anyone debugging an unexpected pipeline behavior who needs to know the exact current call order rather than the documented intent.

### 4. `domain_boundary_audit.generated.json`

**What it is:** A machine-checkable table, one row per `intelligence.*` Postgres table, listing: which domain class's docblock claims ownership, which TypeScript files actually issue a `.schema('intelligence').from('<table>')` call (read or write), and a computed `violation: true/false` flag when a file outside the claimed-owner domain writes to that table.

**Why IntelligenceOS specifically needs this, and needs it as JSON rather than prose:** Architectural Rule 1 — one domain, one writer — is the single most safety-critical rule in this codebase, and it already has a known, documented violation (`HypothesisEngine`, `LearningValidator`, `ProfileBuilder`, and `KnowledgeProcessor` writing directly past their owning domains — see Gap Analysis G-2). A rule that's currently enforced by "hopefully someone notices in code review" needs a mechanical check, not more prose. JSON output means this can be a CI gate with a simple jq/script check (`fail if any row has violation: true that isn't in an explicitly maintained allowlist`), which is the only way this rule will actually hold as the codebase grows past the size where one person can hold the whole domain-ownership map in their head.

**Generation method:** grep/AST scan for every `.schema('intelligence').from('<table_name>')` call site across the package, paired with a hand-maintained `ownership_map.json` (table → owning domain class — this one small mapping file should be the one piece of this artifact that *is* hand-edited, since "who owns this table" is a judgment call the generator shouldn't infer) that the generator diffs actual call sites against.

**Consumed by:** CI, as a hard gate on every PR touching `domains/`, `pipeline/`, or `knowledge/`; anyone proposing to fix the G-2 violation, as the exact list of call sites that need to move.

### 5. `test_coverage_map.generated.md`

**What it is:** A per-source-file table cross-referencing every file in `src/` against the test file(s) that exercise it, plus the current coverage percentage per file (sourced from the `vitest --coverage` JSON output, not re-derived independently) and a flag for any `src/` file with zero matching test files.

**Why IntelligenceOS specifically needs this:** the bootstrap's Testing Philosophy (§13) states that unit tests should mirror pipeline structure and that new pipeline components should get matching test files in the matching subfolder — a structural convention that's easy to follow by habit and easy to silently break once a codebase has multiple contributors. The current coverage thresholds are also stale (40%/30%, dated to "Sprint 0" — Gap Analysis G-4); this artifact is what makes "stale" a checkable, specific claim ("these 6 files in `knowledge/` have 0% coverage and no matching test file") instead of a vague one.

**Generation method:** parse `vitest run --coverage --reporter=json` output, cross-reference file paths against `tests/unit/<pipeline>/` and `tests/integration/` naming conventions, flag mismatches.

**Consumed by:** anyone deciding what to test next; CI, as the actual enforcement mechanism once Gap Analysis G-4's threshold update lands — this artifact's per-file detail is what makes a single repo-wide percentage threshold meaningful rather than something a few well-tested files can paper over for many untested ones.

## What's deliberately not on this list

- **A dependency-impact / blast-radius graph** (the kind the planning documents in `foundations/` reference from a different, much larger system this one was once compared against). With two packages and one clean internal dependency direction (`intelligence-os` depends on `shared-intelligence-types`; nothing depends the other way), a full blast-radius scoring system is solving a problem this repository doesn't have yet. Revisit this if the package count grows past three or four.
- **A database-context dump of live data** (row counts, sample rows). IntelligenceOS's test suite never touches a live database (Testing Philosophy, bootstrap §13) and the schema itself is small and fully captured by `schema.sql` plus `domain_boundary_audit.generated.json` above. A live-data snapshot would be a third source of truth to keep in sync for no corresponding benefit at this repository's current size.
- **An auto-generated architecture diagram beyond what's described in the bootstrap.** The three-pipeline, six-domain shape is small enough to hold in a hand-drawn diagram that changes rarely; `pipeline_topology.generated.md` already covers the part of this that actually drifts (call order inside the pipelines), so a second, broader auto-generated diagram would mostly duplicate the bootstrap's existing hand-drawn one.

## Generation cadence

- **On every PR that touches `packages/`:** CI runs `pnpm context:generate` and fails the build if the regenerated output differs from what's committed. This is the mechanism that keeps these five artifacts from drifting the way the bootstrap and `AGENT_CONTEXT.md` files can only drift slowly (because they require a human to notice).
- **Locally:** any contributor or agent making a structural change runs `pnpm context:generate` before opening a PR, the same way they'd run `pnpm typecheck`.
- **`domain_boundary_audit.generated.json`'s hand-maintained `ownership_map.json`** changes only when an actual domain-ownership decision is made (a new table is added, or ownership of an existing table is deliberately reassigned) — this should require the same review scrutiny as a change to Architectural Rule 1 itself, since it *is* that rule, operationalized.
