# IntelligenceOS — Bootstrap

**This is the canonical onboarding document for IntelligenceOS.** Read it before touching any code. It is written so that a new engineer or a new AI agent, with access to this repository alone, can become productive without asking anyone a question.

> **Maintenance note:** This document describes structure, ownership, and rules — things that change only when an architectural decision is made. It deliberately does not restate information that is auto-derivable from the source tree (exact file lists, line counts, dependency graphs). Those live in `.context/*.generated.md` (see [Repository Context Strategy](./REPOSITORY_CONTEXT_STRATEGY.md)) and are regenerated by tooling, not hand-edited. If you find this document and the generated context disagreeing, the generated context is more likely to be current — but both should be fixed.

---

## 1. Mission

IntelligenceOS turns a user's accumulated context — what they've said, written, uploaded, accepted, and corrected — into a structured set of generation instructions (a **Blueprint**) that a downstream artifact-generation system can consume directly.

Concretely, IntelligenceOS exists to answer one question well: *given everything we've validated about this user, what should the next artifact look like before a single word of it is written?*

It does this through two narrow, stable promises to whatever system calls it:

1. **Before generation:** call `buildBlueprint(request)` and get back a fully-formed `ArtifactBlueprint` — sections, structure, voice, vocabulary, audience calibration, compliance constraints, and any conflicts that were detected and resolved along the way. This call always succeeds, even for a user IntelligenceOS knows nothing about yet.
2. **After delivery:** call `recordFeedbackEvent(event)` to report what happened (accepted, edited, rejected, deployed, or explicitly critiqued). This closes the loop — every recorded event is a chance for the system to get measurably better next time.

Everything else — signal extraction, hypothesis accumulation, learning validation, profile rebuilding, conflict resolution, knowledge ingestion — is an internal concern. The calling system never needs to know any of those names exist.

## 2. Vision

IntelligenceOS is built to be the **single, canonical intelligence substrate** for any system that generates artifacts on a user's behalf — documents, posts, proposals, updates, or any other structured written output. The architecture is deliberately user-centric rather than tied to any one calling application: a user's accumulated intelligence (their voice, goals, expertise, constraints, vocabulary) is a durable asset that should compound in value the more it's used, independent of which generation system happens to be consuming it this quarter.

The system is designed to grow along three deliberately separated axes, each with its own activation trigger rather than a fixed calendar date:

- **Depth of intelligence per user** — from a flat default experience for a brand-new user, through a validated multi-dimensional profile, to (eventually) anticipatory generation that doesn't wait to be asked.
- **Breadth of context scope** — from user-only intelligence, to project-scoped intelligence, to workspace/team-scoped governance, to named-relationship-level audience calibration.
- **Confidence in what's learned** — every piece of intelligence enters as a low-confidence hypothesis and only becomes a trusted `Learning` after it survives a corroboration gate. Nothing is ever assumed correct just because it was inferred once.

A non-goal worth stating explicitly: IntelligenceOS does not try to be maximally clever on day one. The architecture consistently chooses "ship a real, bounded capability now, with an honest stub and a documented activation trigger for what comes later" over "half-build everything." That pattern — visible throughout the domain layer (see §5) — is intentional and should be preserved by future contributors, not "fixed" by filling in every stub at once.

### A note on history

IntelligenceOS was originally conceived as a companion system to an existing product's brand-intelligence runtime, with an eventual integration path in mind. That context shaped some naming the code carried for a while — event types prefixed `brandos.*`, an npm scope of `@brandos/*` — but, per the Gap Analysis (deliverable 5) tracking this as scoped, concrete work, it has since been cleaned up: as of Epic 2 (Platform Publication), the npm scope is `@intelligence-os/*` and every event type shares the `intelligence.*` namespace. Comments that say "Source: BrandOS_..." still exist, deliberately — they're accurate citations to the foundations documents this system's design came from, not a sign the cleanup is incomplete (see Gap Analysis G-7). **Treat IntelligenceOS as a fully independent platform** — that's no longer just the goal; it's the current naming, too.

## 3. Core Concepts

These eleven terms recur throughout the codebase. Knowing them precisely will make every file you open legible on the first pass.

| Concept | Definition |
|---|---|
| **Signal** | A single raw, unvalidated observation extracted from an event (a feedback event, an upload, a conversation). Transient — never persisted to its own table; consumed immediately by the next stage. |
| **Observation** | A Signal after source-quality scoring and taxonomy classification. Carries a *confidence ceiling* determined by where it came from (an explicit statement caps higher than a single inferred behavior). |
| **Hypothesis** | An accumulating, not-yet-trusted proposition built from one or more Observations. Lives in `PROVISIONAL` or `ACCUMULATING` state until it either earns enough corroboration to become a `Learning`, or is discarded/rejected. |
| **Learning** | A validated, trusted atom of intelligence about a user, project, or domain. Has a `stabilityClass` (`permanent` / `long_term` / `medium_term`) that determines how readily it can decay, and a `taxonomyCategory` (one of 25 — see §6) that determines what kind of fact it is. |
| **Intelligence Profile** | The versioned, rebuilt-on-demand synthesis of a user's active Learnings into a single queryable object: voice, goals, constraints, preferences, expertise, vocabulary. Every rebuild increments the version and supersedes the previous one. |
| **Archetype** | A classification of *who the user professionally is* (founder, engineering leader, consultant, researcher, ...). Drives which structural and narrative defaults apply before any user-specific calibration exists. |
| **Knowledge Asset** | A document, playbook, framework, methodology, template, or reference the user (or a project, or a workspace) uploads. Run through its own extraction pipeline (vocabulary, frameworks, structural patterns) independent of the Learning Pipeline. |
| **Artifact Pattern** | A structural template for a type of generated artifact, at one of three levels of specificity: `universal` (everyone starts here), `archetype` (refined for a class of user), `user_calibrated` (refined for one specific user once they've accepted ≥2 artifacts of that type). |
| **Artifact Blueprint** | The single object `buildBlueprint()` returns. Everything the generation system needs: section structure, narrative frame, voice/vocabulary directives, audience calibration, compliance requirements, and a transparent record of any conflicts that were detected and how they were resolved. |
| **Conflict** | A structural tension between two intelligence sources that disagree about how an artifact should sound or what it should contain (e.g., the user's usual register vs. what a board audience expects). Conflicts are *expected*, not exceptional — every blueprint assembly run checks for them and resolves them via a fixed authority ordering. |
| **Domain** | One of six bounded ownership areas (User, Project, Artifact, Knowledge, Relationship, Workspace Intelligence), each implemented as exactly one TypeScript class that is the only code permitted to read or write its tables. See §5 and §10. |

## 5. Package Responsibilities

### `@intelligence-os/shared-types` (renamed from `@brandos/shared-intelligence-types` at Epic 2 — see Gap Analysis G-1)

Owns the contract types that cross the integration boundary: what a request into IntelligenceOS looks like (`ArtifactRequest`), what comes back (`ArtifactBlueprint`), and what feedback looks like going in (`FeedbackEvent`). This package has zero runtime logic — types only. Several sub-shapes (`NarrativeFrame`, `DepthSpecification`, `ComplianceRequirement`) are explicitly marked as placeholders with an escape-hatch index signature (`[key: string]: unknown`) because they were referenced by name in planning material before their owning component existed. They have since been filled in by their owning components (`NarrativePlanner`, `StructurePlanner`, `ConflictResolutionModel` respectively) — the placeholder comments are historical and should be cleaned up (Gap Analysis G-7), but the shapes themselves are real and stable.

### `@intelligence-os/core` (renamed from `@brandos/intelligence-os` at Epic 2 — see Gap Analysis G-1)

Owns everything else. Internally it is seven sub-areas, each described below (`compat/` was added at Epic 2 — Platform Publication; the original six are unchanged). The one rule that holds across all of them: **`IntelligenceOS.ts` is the only file a calling system should construct.** Every other class in this package is reachable only through it, directly or via the pipelines it wires together.

#### `domains/` — the six Domain Stores

Each domain is a class that owns one set of tables and is the *only* code allowed to read or write them. This is the hardest boundary rule in the codebase (see §10) and the one most worth understanding before writing anything new.

| Domain | Owns (tables) | Status |
|---|---|---|
| `UserIntelligenceDomain` | `profiles`, `learnings`, `archetypes`, generic rows in `audience_profiles` | Active — reads fully live; writes (`upsertProfile`, `insertLearning`) are implemented but called only from inside the Learning Pipeline, never from outside it |
| `ProjectIntelligenceDomain` | `projects` | Active — both reads and writes live |
| `ArtifactIntelligenceDomain` | `artifact_patterns`, `artifact_exemplars`, `feedback_events`, `artifact_blueprints` | Active — pattern reads and feedback/blueprint writes live; exemplar promotion is the one remaining stub |
| `KnowledgeIntelligenceDomain` | `knowledge_assets` | Active for reads; see Gap Analysis G-2 for a write-path inconsistency worth knowing about before you touch this domain |
| `WorkspaceIntelligenceDomain` | workspace-scoped knowledge assets, compliance constraints | Partial by design — Phase 1 returns an empty compliance set; full multi-user governance is an explicit Phase 2 capability, not an oversight |
| `RelationshipIntelligenceDomain` | `relationships`, named rows in `audience_profiles` | Deliberately inert — every method throws `DomainNotActivatedError`. Activates when a user has ≥3 external artifacts with named recipients, or via an explicit onboarding signal. Nothing else in the codebase calls into this domain; don't be alarmed that it does nothing yet. |

#### `pipeline/` — the Learning Pipeline

Six classes, wired together by exactly one orchestrator (`FeedbackProcessor`), implementing the Signal → Observation → Hypothesis → Learning → Profile flow described in §8. This is background processing — it runs off the event bus after a `recordFeedbackEvent()` call, never synchronously during blueprint assembly.

#### `blueprint/` — the Blueprint Pipeline

Five classes plus two internal helpers, implementing the assembly flow described in §9. `BlueprintBuilder` is the orchestrator; everything in `blueprint/internal/` is private to this pipeline.

#### `knowledge/` — the Knowledge Pipeline

Six classes implementing upload → normalize → extract (vocabulary / frameworks / structural patterns) → validate → persist, orchestrated by `KnowledgeProcessor`. The newest of the three pipelines (it's what "Sprint 3" in code comments refers to — see §12).

#### `events/` — the Event Bus

One interface (`IntelligenceEventBus`) and one production-grade default implementation (`InProcessEventBus`: synchronous, in-memory, fire-and-forget with per-handler error isolation). This is the seam the calling system uses to observe pipeline milestones if it wants to (it never has to). Two commented-out stub classes (`BullMQEventBus`, `InngestEventBus`) sketch what a distributed swap-in would look like; neither is implemented, and neither is needed until IntelligenceOS runs across more than one process.

#### `compat/` — the Provider Adapter (added at Epic 2)

One class, `IntelligenceOSProvider`: a thin adapter implementing `IIntelligenceProvider` (defined one level up, at the package root — see §5's "one rule" above, since this is the one other file a calling system is expected to construct directly, as an alternative to `IntelligenceOS.ts` rather than something built on top of it) over an injected `IntelligenceOS` instance. Exists for calling systems that want to depend on an interface for dependency injection rather than a concrete class. Deliberately stays thin — see `compat/AGENT_CONTEXT.md` for the rule that this directory must never grow business logic of its own.

#### `types/` — internal type definitions

`entities.ts` (16 of the 24 logical entities — see §6 for why not all 24), `domains.ts` (domain-internal input/filter types), `events.ts` (the 14-member event type union and its 9 payload contracts — 7 from the original architecture doc, plus `LearningReviewedPayload` at E1-1 and `BlueprintBuiltPayload` at Epic 2). Not all of this is re-exported from the package's public `index.ts` — only what a calling system genuinely needs (see `src/index.ts` for the exact list, or `docs/EPIC2_PUBLIC_PLATFORM_SURFACE.md` for the maintained, annotated version of that same list).

#### `db/`

`schema.sql` is the single source of truth for the Postgres schema: 13 tables, all under the `intelligence` Postgres schema (never `public`), with row-level security policies and seed data for the five universal artifact patterns. `db/queries/` is a set of six empty placeholder files awaiting query-builder extraction "when complexity warrants it" — as of this writing, none of them have been activated; see Gap Analysis G-5 for whether that folder should exist at all in its current form.

## 6. Domain Model

IntelligenceOS's logical entity model defines 24 first-class entities. 16 of them have concrete TypeScript shapes in `types/entities.ts`; the rest are either owned by another package (`shared-types`), embedded as JSONB sub-structures rather than standalone tables, or are pipeline-internal and transient. The table below is the map — use it before you go looking for a type that "should" exist.

| Entity | Where it lives | Why |
|---|---|---|
| `IntelligenceProfile`, `Archetype`, `Learning`, `Hypothesis`, `Signal`, `ArtifactPattern`, `ArtifactExemplar`, `KnowledgeAsset`, `Project`, `Relationship`, `AudienceProfile`, `WorkspaceContext`, `FeedbackEventRecord` | `types/entities.ts` | Persisted entities with concrete tables |
| `ArtifactBlueprint`, `FeedbackEvent` (input shape) | `shared-intelligence-types` package | Cross the integration boundary; owned by the contract package, not the engine |
| `User`, `Artifact` | Referenced by ID only (`UserId`, `ArtifactId` type aliases) | Owned externally — `User` by `auth.users`, `Artifact` by whatever generation system calls IntelligenceOS |
| `Goal`, `Constraint`, `Preference`, `Framework`, `Operating Principle`, `Vocabulary Model` | Embedded JSONB sub-structures on `IntelligenceProfile` / `Project` | Owned by `ProfileBuilder`'s internal structuring logic, not standalone — see §7 |
| `Observation` | `pipeline/types.ts`, transient | Pipeline-internal; never persisted, never reaches the public API |
| `Conflict` | Modeled only as `DetectedConflict` / `ConflictResolution` inside `ArtifactBlueprint` | Ephemeral per the Phase 1 contract — not yet a standalone persisted entity (Phase 2 introduces persistent conflict records) |

### Taxonomy

Every `Learning` carries a `taxonomyCategory` — one of exactly 25 fixed string values, from `professional_identity` down to `personal_brand_signal` (full list in `types/entities.ts`). Each category has a `stabilityClass` (`permanent` | `long_term` | `medium_term`) that determines decay behavior and the corroboration threshold required to promote a `Hypothesis` into a trusted `Learning`. Do not add a 26th category casually — every category has a downstream weight in `ProfileBuilder`'s composite-confidence scoring and a corroboration requirement in `HypothesisEngine`; adding one means updating both.

**On visual modality (color, typography, layout, mood, motion):** these are not a separate domain and never have been — they're style/identity/structure/governance facts expressed through a visual channel instead of a verbal one, and they belong inside the existing six domains exactly the way text-voice facts do (`personal_brand_signal` or a similarly-scoped taxonomy value under `UserIntelligenceDomain` for style/mood, `WorkspaceIntelligenceDomain`'s compliance constraints for declared visual governance, `ArtifactPattern`/`ArtifactExemplar` for structural defaults, `KnowledgeAsset` for uploaded reference material like brand guidelines or design systems). See `ADR-001-VISUAL-INTELLIGENCE.md` for the full reasoning — the short version is that a seventh domain organized by subject matter (visual vs. textual) rather than by ownership and lifecycle would duplicate machinery (confidence, decay, conflict resolution) the six existing domains already provide.

### Archetypes

`ArchetypeType` is an open string union (16 named values plus `| string` as an escape hatch). It is deliberately not closed: archetype classification is expected to grow as real user data comes in, and a closed union would force a breaking type change for every new archetype discovered.

## 7. Learning Pipeline

The path from "the user did something" to "the system durably knows something new about them."

```
FeedbackEvent (or other input)
       │
       ▼
SignalExtractor        — classify source, apply the quarantine gate (see below), emit 0..N Signals
       │
       ▼
ObservationBuilder      — score source quality → assign a confidence ceiling, attach taxonomy category
       │
       ▼
HypothesisEngine        — match against existing Hypotheses (corroborate / contradict) or create new ones;
       │                  drive the state machine: PROVISIONAL → ACCUMULATING → VALIDATED / DISCARDED / REJECTED
       ▼
LearningValidator        — check corroboration threshold + unresolved contradictions; promote to a Learning
       │                   (state: VALIDATED), assign stabilityClass + decayRate
       ▼
ProfileBuilder           — decide whether the accumulated changes warrant a profile rebuild; if so, version
                            the profile, recompute composite confidence, emit `intelligence.profile.updated`
```

**Orchestration.** `FeedbackProcessor` is the only place these six classes are wired together. It subscribes to `brandos.artifact.feedback` on the event bus and runs every signal produced by an event through the full pipeline, never letting one stage's failure stop the others — failures are collected into `PipelineRunResult.errors` and the run continues. This graceful-degradation behavior is load-bearing: a new user with zero profile, a feedback event with no project, or one failing extractor must never abort the whole pipeline run.

**The quarantine gate.** Before anything is extracted, every signal is checked against three flags: `role_play`, `hypothetical`, `emotional_state`. A signal carrying any of these is discarded immediately unless it carries an explicit identity declaration overriding the flag. This is the single most important safety property of the Learning Pipeline: a user adopting a persona for a task, speculating about a hypothetical, or venting in the moment must never be mistaken for a durable statement about who they are. Get this gate wrong and every downstream Learning becomes untrustworthy.

**Confidence ceilings.** A `Signal`'s source determines the maximum confidence any `Observation` built from it can ever carry, regardless of how many times it's corroborated:

| Source quality | Ceiling |
|---|---|
| `explicit_statement` | 1.00 |
| `demonstrated_behavior` | 0.90 |
| `uploaded_artifact` | 0.90 |
| `inferred` | 0.35 |

**Corroboration thresholds** (how many independent observations a Hypothesis needs before `LearningValidator` will promote it), by stability class:

| Stability class | Required corroborations |
|---|---|
| `permanent` | 2 |
| `long_term` | 3 |
| `medium_term` | 2 |

**Correction override.** A `brandos.user.correction` event is the one input that bypasses every gate above. An explicit user correction is treated as the highest-authority signal in the system and applies immediately as a `CONFIRMED` learning — no corroboration wait, no quarantine check. This is intentional: if the user tells you directly that something is wrong, believe them immediately.

**Why Signals and Hypotheses aren't persisted to their own pipeline-managed tables outside of what the schema defines:** `intelligence.signals` and `intelligence.hypotheses` *do* exist as real tables in `schema.sql`, but the current `SignalExtractor` implementation deliberately keeps Signals in-memory and passes them straight to `ObservationBuilder` rather than writing them — documented in the file itself as a Sprint 2 scope decision to avoid adding a new domain-store write path before it was load-bearing. Worth knowing if you're debugging why `intelligence.signals` stays empty in a live database: that's expected with the current implementation, not a bug.

## 8. Blueprint Pipeline

The path from "the generation system is about to create an artifact" to "here is everything it needs to know to do that well."

```
ArtifactRequest
       │
       ▼ (Step 1 — parallel fetch, every call independently fails-soft to null/[])
   ┌───────────────┬──────────────────┬───────────────────────┬──────────────────────┐
   │ current        │ current archetype │ ProjectContextBuilder  │ AudienceCalibrator    │
   │ profile        │                   │ (project + workspace   │ (generic audience    │
   │                │                   │  + project-scoped      │  profile, falls back  │
   │                │                   │  learnings + knowledge │  to system defaults)  │
   │                │                   │  assets)               │                       │
   └───────────────┴──────────────────┴───────────────────────┴──────────────────────┘
       │
       ▼ Step 2 — StructurePlanner: pick sections + depth
         (priority: user_calibrated pattern → archetype pattern → universal pattern → FALLBACK_SECTIONS)
       │
       ▼ Step 3 — NarrativePlanner: assemble voice + vocabulary directives (synchronous, pre-loaded data only)
       │
       ▼ Step 4 — detectConflicts(): compare what the user's intelligence prefers against
         what workspace/audience/project actually require
       │
       ▼ Step 5 — ConflictResolutionModel: resolve every detected conflict via the fixed
         authority ordering (§10)
       │
       ▼ Step 6 — extract compliance requirements from workspace context
       │
       ▼ Step 7 — assemble the ArtifactBlueprint (stable id, full audit trail of what was
         detected and how it was resolved)
       │
       ▼ Step 8 — persist + emit `intelligence.blueprint.built` — both fire-and-forget;
         the blueprint is returned to the caller even if either fails
```

**The one guarantee that overrides everything else in this pipeline:** `buildBlueprint()` must always succeed and always return a usable blueprint — for a brand-new user with zero stored intelligence, for a request with no `projectId` or `workspaceId`, for an artifact type that isn't one of the five seeded universal patterns. Every step above is written to degrade to a documented default rather than throw. If you're adding a new step to this pipeline, this guarantee is the first thing to preserve, not an afterthought to bolt on later.

**Authority ordering** (Step 5's resolution rules, highest to lowest):

```
USER_CORRECTION (10) > EXPLICIT_INSTRUCTION (9) > USER_ESTABLISHED_PATTERN (8)
  > WORKSPACE_COMPLIANCE (7) > PROJECT_CONTEXT (6) > AUDIENCE_CALIBRATION (5)
  > ARCHETYPE_INTELLIGENCE (4) > UNIVERSAL_PATTERN (3) > SYSTEM_DEFAULT (2)
```

Four named rules sit on top of this ordering and are checked in this sequence:

1. **COMPLIANCE** — a workspace compliance requirement always wins, full stop. This is the *Immutability Rule*: no other authority, including an explicit user correction, can override it.
2. **WORKSPACE** — for non-compliance conflicts where one side is workspace-scoped, workspace still wins for general governance.
3. **RECIPIENT** — for register conflicts between the user's usual voice and what an audience expects, the audience wins. The artifact has a reader; their context outranks the sender's habitual preference.
4. **PROJECT** — for vocabulary conflicts, the project's vocabulary model wins over the user's general vocabulary for project-scoped content.

Anything else falls through to plain authority-level comparison. When a resolution represents a *significant* departure from what the user would normally expect, a human-readable transparency note is attached to the resolution (the **Transparency Rule**) — this is what eventually surfaces to the user as "here's why this came out differently than your usual style."

**Knowledge Pipeline, briefly:** runs independently of the above two (upload → normalize → extract vocabulary/frameworks/structural patterns → validate confidence → persist), feeding `KnowledgeAsset` rows that `ProjectContextBuilder` and `NarrativePlanner` read from. It is heuristic-only by design — no LLM calls inside any extractor — which keeps it deterministic and fast at the cost of being less semantically sophisticated than an LLM-based extractor would be. That trade-off is intentional for Phase 1 and documented in each extractor's header comment.

## 9. Read Order

Read these in order. Each step assumes only what came before it.

1. **This document, in full**, before anything else.
2. **`packages/shared-intelligence-types/src/`** (all four files — they're short). This is the entire vocabulary the outside world uses to talk to IntelligenceOS; nothing else makes sense without it.
3. **`packages/intelligence-os/src/IntelligenceOS.ts`**. The whole system through its one entry point — six public methods (matching `IIntelligenceProvider`, which it formally `implements` as of Epic 2), six domains wired into a constructor, two pipelines registered as event subscribers.
4. **`packages/intelligence-os/src/errors.ts`**. Five classes, five minutes. Knowing `PhaseNotImplementedError` vs. `DomainNotActivatedError` vs. `DatabaseError` up front means every stub you encounter later reads as "documented, deliberate gap" rather than "broken code."
5. **`packages/intelligence-os/src/types/entities.ts`**, then **`types/domains.ts`**, then **`types/events.ts`**. The shape of everything the system persists or passes around internally.
6. **`packages/intelligence-os/src/db/schema.sql`**, top to bottom, including the comments. This file is more readable than it looks — every table has a one-line rationale, and the RLS section at the bottom explains the one non-obvious policy fix (`artifact_patterns`) that the architecture doc itself got wrong on the first pass.
7. **`packages/intelligence-os/src/domains/`** — read `UserIntelligenceDomain.ts` first (it's the richest and best-commented), then skim the other five. Pay attention to which methods are real and which throw `PhaseNotImplementedError` / `DomainNotActivatedError` — the header docblock of every domain file tells you this up front.
8. **`packages/intelligence-os/src/pipeline/`** in this order: `types.ts` → `SignalExtractor.ts` → `ObservationBuilder.ts` → `HypothesisEngine.ts` → `LearningValidator.ts` → `ProfileBuilder.ts` → `FeedbackProcessor.ts` last, since it's the orchestrator and only makes sense once you know what it's wiring together.
9. **`packages/intelligence-os/src/blueprint/`** in this order: `internal/defaults.ts` → `internal/conflictDetection.ts` → `ProjectContextBuilder.ts` → `AudienceCalibrator.ts` → `StructurePlanner.ts` → `NarrativePlanner.ts` → `ConflictResolutionModel.ts` → `BlueprintBuilder.ts` last.
10. **`packages/intelligence-os/src/knowledge/`** in this order: `types.ts` → `KnowledgeAssetExtractor.ts` → `VocabularyExtractor.ts` → `FrameworkExtractor.ts` → `PatternExtractor.ts` → `KnowledgeValidator.ts` → `KnowledgeProcessor.ts` last.
11. **`packages/intelligence-os/src/events/IntelligenceEventBus.ts`**. Short, and ties §7–10 together once you've seen what emits and subscribes to what.
12. **`packages/intelligence-os/tests/integration/intelligence-os.test.ts`**. The fastest way to see the whole system exercised end-to-end, including exactly what a mocked Supabase client needs to support.

## 10. Architectural Rules

These are the rules a pull request can be rejected for violating. Each is load-bearing for a specific failure mode the architecture is designed to prevent.

1. **One domain, one set of tables, one writer.** Each of the six domain classes owns a disjoint set of `intelligence.*` tables. No domain may write to another domain's tables, and no code outside the domains layer may write to `intelligence.*` directly. *(Current exception, tracked as a real gap, not a precedent to extend: `HypothesisEngine`, `LearningValidator`, `ProfileBuilder`, and `KnowledgeProcessor` currently hold a raw Supabase client and write to `intelligence.hypotheses`, `intelligence.learnings`, `intelligence.profiles`, and `intelligence.knowledge_assets` directly, bypassing `UserIntelligenceDomain` and `KnowledgeIntelligenceDomain` respectively, even though those domains define the corresponding entity types. See Gap Analysis G-2. Do not use this as justification for a new direct-write path — fix it toward the domain, not away from it.)*
2. **All Blueprint-assembly intelligence access goes through domain APIs.** `BlueprintBuilder`, `ProjectContextBuilder`, `StructurePlanner`, `AudienceCalibrator`, and `NarrativePlanner` are explicitly forbidden from issuing direct Supabase queries — stated in their own header comments. If a blueprint-assembly component needs new data, add a method to the owning domain; don't reach around it.
3. **`buildBlueprint()` never throws for missing data.** Every intelligence fetch inside blueprint assembly is wrapped in a fail-soft fallback (`.catch(() => null)` or equivalent). A blueprint must always be returned. If you're adding a new data source to blueprint assembly, its absence must degrade to a documented default, not an exception.
4. **The Learning Pipeline runs off the event bus, never synchronously inside request handling.** `recordFeedbackEvent()` persists immediately and emits an event; it does not await pipeline completion. Blueprint assembly reads only from already-processed, already-validated Learnings — it never triggers pipeline processing as a side effect of being read.
5. **Quarantine and corroboration gates are not optional, configurable, or bypassable** except by an explicit user correction (§7). Don't add a parameter that lets a caller skip the quarantine check "just this once" — that defeats the property the gate exists to guarantee.
6. **Phase boundaries are activation triggers, not arbitrary flags.** `RelationshipIntelligenceDomain` is fully stubbed not because it's unfinished work sitting in a backlog, but because its activation condition (≥3 external artifacts with named recipients) hasn't been met by the system's design yet. Don't "turn on" a Phase 2/3 capability without first checking whether its documented trigger condition is the right one to honor — the gating exists on purpose.
7. **The public API surface is `IIntelligenceProvider`'s six methods on `IntelligenceOS`, plus the bus accessor.** `buildBlueprint`, `recordFeedbackEvent`, `ingestKnowledgeAsset`, `upsertProject`, `reviewLearning`, `getBrandSummary`, and `eventBus`. As of Epic 2, this is formalized as the published `IIntelligenceProvider` interface (`IntelligenceOS implements` it directly) rather than just a header comment's claim — treat any change to it as a deliberate, reviewed, versioned decision, not an incidental side effect of adding a feature. See `IIntelligenceProvider.ts`'s docblock for why `eventBus` is the one method deliberately excluded from the formal interface.
8. **Compliance constraints are immutable by anyone except a workspace admin**, and once detected as a conflict, compliance always wins resolution (the Immutability Rule, §8). No Learning, no user correction, no audience calibration may override a compliance requirement.
9. **Taxonomy categories and stability classes are closed, coordinated unions.** `TaxonomyCategory` (25 values), `StabilityClass` (3 values), and `DecayRate` (4 values) each have downstream consumers in multiple files (`ObservationBuilder`'s stability map, `ProfileBuilder`'s weight table, `HypothesisEngine`'s corroboration thresholds). Adding or removing a value means updating every consumer, not just the type definition.
10. **Every domain method's deferred/stub status is documented in its own docblock**, using the `✓` (real) / `✗` (stub) convention already established in every domain file. If you implement a stubbed method, update the docblock in the same commit — don't leave it claiming `✗` for something that now works.

## 11. Development Workflow

```bash
# From the workspace root:
pnpm install                      # installs across both packages
pnpm typecheck                    # tsc --noEmit, recursive across packages
pnpm test                         # vitest run, recursive across packages
pnpm test:coverage                # vitest run --coverage
pnpm clean                        # rm -rf dist, recursive
```

Tests require no live infrastructure. Every Supabase interaction in every test file is mocked via a small factory (see `createMockSupabase` in `tests/integration/intelligence-os.test.ts`) that chains `.schema().from().select().eq().maybeSingle()` and friends. You do not need a Supabase project, a `.env` file, or network access to run the full test suite — this is intentional and should stay true as the system grows.

To stand up a real database (for manual testing or a staging deployment, not for the automated test suite): apply `packages/intelligence-os/src/db/schema.sql` once against a fresh Supabase Postgres instance, then run the `GRANT`/`ALTER DEFAULT PRIVILEGES` statements documented in the comment block at the top of that file. The `intelligence` schema must be added to Supabase's exposed-schemas list before the client can query it.

There is currently no ESLint configuration or CI workflow definition in this repository — see Gap Analysis G-8 for what's recommended. The one mechanical boundary check that does exist is `packages/intelligence-os/scripts/check-boundaries.mjs`, runnable via `pnpm --filter @intelligence-os/core run check:boundaries` — it enforces RULE-IOS-ISOLATION and RULE-SIT-ISOLATION (see §10 and `docs/IMPLEMENTATION_STATUS.md`'s Platform Boundary Rules section). It is not ESLint-based — it is a standalone Node script that statically validates import specifiers. A root-level `README.md` and `.gitignore` were added at Epic 2 (Gap Analysis G-3, now resolved).

## 12. Implementation Philosophy

A handful of consistent decisions run through every pipeline in this codebase. Recognizing them will save you from re-deriving "is this a bug?" every time you hit one of these patterns:

- **"Smallest implementation that satisfies the contract."** Repeatedly, when a design choice has a simple version and a more complete version, the simple version ships first with an explicit comment naming what was deferred and why (e.g., Signals staying in-memory instead of being persisted — §7). This is not corner-cutting; it's a stated policy, visible in code comments across `SignalExtractor`, `KnowledgeAssetExtractor`, and elsewhere.
- **Heuristic, not LLM-based, pipeline internals.** Every extractor in the Knowledge Pipeline and every classifier in the Learning Pipeline is deterministic pattern-matching — no calls to a language model anywhere inside pipeline internals. This keeps the system fast, testable without mocking an AI provider, and fully deterministic for a given input. If you're tempted to add an LLM call inside an extractor, that's a significant architectural change, not an incremental improvement — flag it for design review rather than slipping it in.
- **Confidence is earned, never assumed.** Nothing becomes a trusted `Learning` on a single observation (except an explicit user correction). Nothing becomes a `user_calibrated` artifact pattern on a single accepted artifact (the threshold is ≥2). This conservatism is the system's primary defense against confidently-wrong personalization — a model that's certain but wrong produces worse outcomes than a model that knows it doesn't know yet (this principle is stated explicitly in the foundational taxonomy document, §13, Section H.1).
- **Graceful degradation is a tested property, not an incidental behavior.** Both the Learning Pipeline (`FeedbackProcessor.process()`) and the Blueprint Pipeline (`BlueprintBuilder.build()`) are written so that a failure in one stage doesn't abort the others. This is exercised directly in the test suite, not just asserted in comments.
- **Documentation lives next to the code it describes, with provenance.** Nearly every file in this codebase opens with a docblock stating what it's responsible for, what's real vs. stubbed, and (in the current codebase) a "Source:" line pointing at the planning document a design decision came from. As IntelligenceOS continues independently, new files should keep the first two habits (responsibility + real/stubbed status) and drop the "Source:" provenance convention in favor of pointing at this bootstrap document or the relevant `AGENT_CONTEXT.md` — see Gap Analysis G-7.

## 13. Testing Philosophy

- **Tests never touch a live database.** Every test, unit or integration, runs against a mocked Supabase client. This is a hard constraint, not a convenience — it's what makes `pnpm test` runnable with zero setup, by a human or an agent, anywhere.
- **Unit tests mirror the pipeline structure.** `tests/unit/pipeline/`, `tests/unit/blueprint/`, `tests/unit/knowledge/` — each pipeline's components are tested in roughly the same dependency order they're read in (§9). If you add a new pipeline component, add its test file in the matching subfolder, not in `integration/`, unless it genuinely needs the full stack.
- **Integration tests exercise the public methods, not internals.** `tests/integration/intelligence-os.test.ts` calls `IntelligenceOS` exactly the way a generation system would — through the public surface — and asserts on observable outcomes (DB calls made, events emitted, errors thrown). `tests/integration/blueprint.test.ts` does the same for the blueprint-assembly path specifically. New integration tests should hold to this same external-only vantage point.
- **Graceful degradation gets explicit test coverage**, not just incidental coverage from happy-path tests. If you add a new fallback path (a new `.catch(() => default)`), write a test that actually exercises the failure branch — coverage tools won't tell you the fallback logic is *correct*, only that it ran.
- **Coverage thresholds are currently stale** (`lines: 40, branches: 30`, dated to "Sprint 0" in the config comment, despite three pipelines' worth of code having shipped since). Don't treat the configured threshold as the bar to clear — treat it as a gap to close. See Gap Analysis G-4.

## A note on `foundations/`

This repository's `foundations/` directory contains the original specification and planning documents IntelligenceOS was built from (architecture spec, formal contracts, logical schema, learning taxonomy, and a prior cross-system adoption analysis). They remain useful as **deep-reference material** — the Contracts document in particular has a phase-by-phase activation model (§J) that's more detailed than what's summarized in §10 above, and the Taxonomy document's exclusion framework (§H) is the authoritative source for *why* the quarantine gate exists.

They are not onboarding material, and they are not always literally accurate against the current code — several documented mismatches exist (see Gap Analysis), and the documents themselves predate parts of the implementation. Treat this bootstrap document, the package `AGENT_CONTEXT.md` files, and the generated context artifacts as authoritative for "how IntelligenceOS works today." Treat `foundations/` as authoritative for "why it was designed this way" when this document doesn't already explain it.
