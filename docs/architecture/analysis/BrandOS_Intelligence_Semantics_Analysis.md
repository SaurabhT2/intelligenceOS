# IntelligenceOS — BrandOS Intelligence Semantics Analysis

> **Historical planning document.** Pre-Epic-1 semantic review of BrandOS's observable intelligence model and how IntelligenceOS should represent equivalent knowledge natively. Findings here drove the E1-4 visual intelligence design (ratified by ADR-001) and the E1-5 classification scheme correction. For current state, see `docs/IMPLEMENTATION_STATUS.md` and `docs/ADR/ADR-001-VISUAL-INTELLIGENCE.md`.

## Executive Summary

This document analyzes BrandOS as an observable intelligence system — what it has learned to know about a brand, why each category of knowledge exists, and how reliable each piece of evidence behind it actually is — and then asks how IntelligenceOS should represent that same knowledge natively, rather than by porting BrandOS's implementation. It strengthens Epic 1 of the existing roadmap; it does not redesign the architecture, the roadmap, or the Implementation Guide, all of which remain authoritative.

**Three findings carry the most weight:**

1. **BrandOS has a real, sophisticated learning model for *style* (Class A/B signals — tone, hook, structure, all topic-abstracted) and almost no model for *substantive, evolvable belief* (Class C — domain positions, market narratives, which accumulate without ever resolving contradictions).** IntelligenceOS already has the exact scaffolding BrandOS is missing — `Hypothesis.requiredCorroborations`/`currentCorroborations`/`highQualityContradictions` — sitting unused. This is the highest-value design opportunity in the analysis.

2. **BrandOS collects the single highest-trust evidence source in its entire system — explicit human satisfaction feedback — and never reads it back.** The `feedback` table is write-only; nothing in the codebase consumes it. IntelligenceOS already has a native home for exactly this evidence type (`ArtifactExemplar`, `FeedbackEvent.eventType: 'explicit_feedback'`), unused for this purpose so far. Closing this loop is low-risk and high-value: no new schema, only a missing consumption path.

3. **A meaningful share of BrandOS's own design intent was never built.** A fully-typed visual/semantic identity system (`ISemanticIdentity`, `IVisualIdentity`, `IdentityVersionRecord`) exists with no live extractor behind most of it; `resolvedVisualIdentity` is threaded through the live runtime and never populated by anything. This is treated as real evidence of *intent* throughout this analysis — what BrandOS believed was worth learning, even though it never shipped the mechanism.

Two corrections to the Implementation Guide surface from this deeper read and are carried into Deliverable 8: the BrandOS-side VLM-to-brand-memory bridge (previously treated as not-yet-built, gated on Epic 3 infrastructure) is **already live** today via the same proxy function used for text artifacts; and the calibration source cited for E1-5's classification thresholds (`BrandSignalClassifier.ts`) does not exist — the real ground truth is the `SIGNAL_CLASSIFICATION` lookup table and the `addSignal()` extraction logic, both fully inventoried below.

**No Epic 1 Feature requires a new task or schedule change.** Three of five features (E1-1, E1-2, E1-4) get a scoped, additive refinement; two (E1-3, E1-5) are confirmed correct as already planned, with one citation fix for E1-5. See Deliverable 8 for the complete, per-feature reasoning.

**Document structure:**
1. Complete Learning Inventory — 36 categories across BrandOS's two live systems (inferred brand memory, declared persona) plus its dormant visual-identity design and confirmed absences
2. Learning Objective Analysis — what, why, and artifact-quality impact per category
3. Evidence Model — every real and orphaned evidence source, with reliability and weighting guidance
4. Knowledge Characteristics — stability, decay, evolution, confirmation, contradiction, and versioning per category
5. IntelligenceOS Domain Mapping — onto IOS's six real domains (User, Workspace, Project, Relationship, Artifact, Knowledge Intelligence)
6. Native IntelligenceOS Design Recommendations — the core deliverable: what each learning category should become in IOS's own vocabulary (`Learning`, `Hypothesis`, `ArtifactPattern`, `ArtifactExemplar`, `KnowledgeAsset`, `AudienceProfile`)
7. Capability Evolution Opportunities — thirteen ways IntelligenceOS can exceed BrandOS, not just match it
8. Epic 1 Recommendations — the specific, scoped refinements this analysis motivates, mapped to the existing Implementation Guide's task structure

# BrandOS Intelligence Semantics Analysis

**Purpose:** Strengthen Epic 1 of the IntelligenceOS Engineering Roadmap by analyzing BrandOS as a *behavioral reference* — what it has learned to know, and why — rather than as a system to be ported. The Adoption Strategy, Engineering Roadmap, and Implementation Guide remain authoritative and unchanged. This document does not redesign them; it gives Epic 1 a semantic foundation so that each implementation task can be justified by a learning objective rather than by feature parity.

**Method:** Every claim below is grounded in BrandOS source actually read for this analysis: `packages/brand-intelligence/src/cognition/brand-memory-service-v2.ts` (the extraction/decay/consolidation engine), `style-projection-resolver.ts` (the read-side projection and conflict resolution), `brand-context.ts` (the declared-voice merge), `packages/contracts/brand-cognition-contracts.v2.ts` and `src/identity-types.ts` (the full type system, including a dormant visual-identity model), `BrandIntelligenceRuntime.ts` (what's actually wired live), the VLM asset-analysis route, the feedback route, and the relevant `.context/database_context.generated.md` table definitions. Where BrandOS's own code comments state intent directly (several do, unusually clearly), those are treated as first-class evidence of *why*, not just *what*.

Two corrections to the existing Implementation Guide surface naturally from this deeper read and are folded into Deliverable 8, not litigated separately: (1) the VLM-to-brand-memory bridge the guide treated as not-yet-built (E1-4) is **already live** in `apps/web/app/api/assets/[id]/analyze/route.ts`, via the same `recordBrandMemoryObservation` proxy used for text artifacts; (2) the calibration source the guide cited for classification thresholds (`BrandSignalClassifier.ts`) does not exist — the real ground truth is the `SIGNAL_CLASSIFICATION` lookup table and the `addSignal()` extraction logic in `brand-memory-service-v2.ts`, both fully inventoried below.

# Deliverable 1 — Complete Learning Inventory

BrandOS learns across two structurally separate systems that this analysis treats as one behavioral whole (IntelligenceOS doesn't need to preserve the separation, but needs to understand both halves of what's actually being learned):

**System A — Inferred Brand Memory** (`@brandos/brand-intelligence`, V2/active): signals extracted automatically from every artifact BrandOS generates, classified, confidence-scored, decayed, and consolidated without a human in the loop unless review is explicitly enabled.

**System B — Declared Persona** (`@brandos/auth`, `personas` table): facts the user states directly about their brand (`tone`, `domain`, `audience`, `key_themes`, `visual_style`) — never inferred, never decayed, edited only by explicit user action.

A third, designed-but-dormant system — **Visual/Semantic Identity** (`ISemanticIdentity`/`IVisualIdentity` in `identity-types.ts`) — exists as fully-specified types with no live extractor behind most of it; it's listed below because its *design intent* is real evidence of what BrandOS believes is worth learning, even though the implementation never shipped.

|---|---|---|---|---|
| 1 | **Tone pattern** | A (Class A) | Phrasal description of voice markers — first-person plural, second-person address, measured enthusiasm, rhetorical questions, long-form vs. punchy | `extractTonePatterns()` |
| 2 | **Hook style** | A (Class A) | Single label per artifact: numeric lead / question hook / bold headline / statement / narrative | `extractHookStyle()` |
| 3 | **Format preference** | A (Class A) | Bullet lists, numbered lists, header sections, bold emphasis, long-form paragraphs | `extractFormatPreferences()` |
| 4 | **Narrative arc** | A (Class A) | Problem→solution, before→after, list-insight, contrarian reframe, data→insight — up to 2 per artifact | `extractNarrativeArc()` |

| 20 | **Artifact visual metadata (designed, dormant)** | C | Layout type, section/heading/word counts, use of cards/data-viz/imagery — typed as observation input, never actually constructed by any caller | `ArtifactVisualMetadata` |
| 21 | **Asset-derived brand signal** | A (text path) | VLM-derived description + mood + recommendations from an uploaded image or document, fed into the *same* Class A/B/C pipeline as generated text | asset-analyze route → `recordBrandMemoryObservation` |
| 22 | **Explicit human satisfaction signal (collected, unused)** | — (orphaned) | Categorical rating (`useful`/`generic`/`off_tone`/`excellent`/`needs_work`) plus free-text note, tied to a specific generated artifact | `feedback` table — write-only, no reader anywhere in the codebase |

| 25 | **Repair status (gating signal)** | gating signal | Boolean flag — was this artifact auto-repaired by governance before being shown to the user; if so, it is excluded from learning entirely | `wasRepaired` gate in `learn()` |
| 26 | **Topic identity (hashed, never stored raw)** | gating signal / evidence weight | SHA-256 of the normalized user topic — used only to compute how many *distinct* topics a pattern has been seen across, then discarded | `hashTopic()` |
| 27 | **Confidence (per-signal)** | derived | Weighted function of frequency (35%), topic diversity (45%), source quality score (20%), capped at 0.95 | `recalculateConfidence()` |
| 28 | **Frequency / recurrence** | derived | Count of artifacts that have produced a given (workspace, type, content) tuple | `IBrandMemorySignal.frequency` |
| 29 | **Topic diversity** | derived | Count of *distinct* topic hashes that have produced a given signal — the actual generalization-strength metric | `IBrandMemorySignal.topic_diversity` |
| 30 | **Style drift** | derived/temporal | Detection that a newer, frequently-seen pattern is displacing an older one not observed in 45+ days | `resolveConflicts()` in `style-projection-resolver.ts` |
| 31 | **Signal decay** | temporal | Time-based confidence reduction for Class A/B signals not reinforced recently; Class C never decays | `runDecay()` |
| 32 | **Signal consolidation** | maintenance | Exact-duplicate and near-duplicate (Levenshtein ≤2) merging, plus per-type capacity enforcement | `runConsolidation()` |
| 33 | **Workspace brand voice (declared, low-fidelity)** | B | Flat tone/cadence/banned-phrases/audience-type, merged from persona + request overrides | `mergeBrandContext()` |
| 34 | **Operating principle: topic-independence** | meta-knowledge | A codified rule (not a learned signal, but knowledge *about* the system itself) that persona-level business identity must never bias topic selection | `TOPIC-DRIFT-FIX-002` comment in `brand-context.ts` |
| 35 | **Edit/revision behavior** | — (does not exist) | No regenerate/edit flow exists in the product; every generation is insert-only | confirmed via `versions/route.ts`'s own doc comment |
| 36 | **Goals / constraints / strategic intent** | — (does not exist) | No representation anywhere in BrandOS; the planner agent does tactical content-idea generation only, with no goal or constraint model | `plannerAgent.ts` |

This inventory deliberately includes rows 19–22 and 35–36 — designed-but-unused capability and confirmed absences — because "what BrandOS is learning" is incomplete without "what BrandOS clearly intended to learn but doesn't yet" and "what BrandOS has never attempted to learn." Both are direct inputs to Deliverable 7.

# Deliverable 2 — Learning Objective Analysis

| Learning Category | What is learned | Why it matters | Artifact Quality Impact |
|---|---|---|---|
| Tone pattern | The recurring *voice* markers of past well-scoring output — point of view, address style, enthusiasm register, structural density | Without it, every artifact defaults to a generic professional voice; with it, output sounds like it came from the same author as the brand's past best work | Directly injected into the system prompt as `tonePatterns` — among the highest-leverage signals, since voice mismatch is the most visible failure mode to a human reader |
| Hook style | Which opening-line strategy this brand's best content actually uses | A brand that always opens with a number shouldn't suddenly open with a rhetorical question; consistency in the first line is disproportionately important to perceived quality | High — first impressions drive read-through; this is a single highest-confidence value injected per generation |
| Format preference | Structural habits — bullets vs. prose, bold emphasis, paragraph length | Matches the *shape* of content to what this brand's audience is used to consuming | Medium — affects scannability and perceived professionalism more than core message quality |
| Narrative arc | Which storytelling shapes this brand's content tends to use successfully | Gives the model a proven shape to fill rather than inventing structure from scratch each time | High — structure is often the difference between content that "reads as branded" and content that reads as generic AI output |
| Executive cadence | Sentence-length rhythm that correlates with this brand's past high-scoring work | A B2B thought-leadership brand and a consumer-facing brand have systematically different rhythms; getting this wrong reads as tonally foreign even if the words are fine | Medium-high — cadence mismatches are subtle but cumulative across a long artifact |
| Argumentation style | How this brand tends to *justify* claims — data, anecdote, example, framework | Determines whether generated reasoning "sounds like" this brand's actual thinking style, not just its vocabulary | High — this is closer to *how the brand thinks* than *how the brand sounds*, and is hard to fake without learned signal |
| Evidence pattern | What kind of proof this brand reaches for | Complements argumentation style with the concrete evidentiary habit (stats vs. quotes vs. anecdote) | Medium |
| CTA intent | The *functional* relationship this brand tries to build with its audience at the end of a piece (community invitation vs. activation vs. conversion), independent of the literal verb used | Lets the model choose a CTA appropriate to context while preserving the brand's *strategic* habit, rather than parroting the exact verb that happened to appear in one past post | Medium-high — a wrong CTA *intent* (asking for a sale when the brand's habit is community-building) is a strategic mismatch, not just a wording one |
| Title pattern | Structural templates for headlines, with topic content abstracted away | Generalizes a brand's title habit across topics it has never written about before | High — titles are read by everyone; getting the shape right even on a brand-new topic is a strong quality signal |
| Structural arc | Topic-free shapes like "N-stage progression" | Lets the model reuse a proven content skeleton for a completely new topic | High — this is precisely the mechanism that should let a brand-voice signal *generalize* rather than only ever working for topics it's already seen |
| Value frame | Verb-led value-proposition templates, domain-abstracted | Captures how this brand frames benefit statements structurally, reusable across any product/topic | Medium |
| Hook pattern | Structural opening templates (e.g., statistic-led with the statistic itself removed) | Same generalization logic as structural arc, applied specifically to the first line | Medium-high |
| Domain theme | Actual recurring topics/subjects this brand writes about | Useful for *understanding* the brand (analytics, persona summaries) but explicitly never reaches generation | Zero direct generation impact by design — its value is informational/strategic, not generative; injecting it would cause topic drift |
| Named framework | This brand's own named methodologies | Same as domain theme — valuable as institutional memory, structurally barred from prompt injection | Zero direct generation impact by design |
| Domain position | Substantive stances the brand has taken | Same — archival, not generative | Zero direct generation impact by design |
| Market narrative | How the brand frames industry trends | Same — archival, not generative | Zero direct generation impact by design |
| Declared identity | What the user *says* their brand is, before any inference has accumulated | Available from message one, with zero cold-start problem — this is the bridge that makes a brand-new workspace usable immediately | High at low signal-volume; its relative importance decreases as System A accumulates real evidence |
| Declared/partial visual identity | Color palette and typography personality merged from VLM analysis into the persona | The only visual signal that currently round-trips back into anything read at generation time | Currently low-to-medium, because nothing in the live text-generation path actually reads `persona.visual_style` for prompt construction (confirmed: `IdentityContributor` reads `styleProjection`, not `persona.visual_style`) — so this is *stored* brand truth without a confirmed *consumer* |
| Visual/semantic identity (dormant) | Intended: color/typography/layout/motion/mood systems at the same rigor as text style | Would let visual-artifact generation (decks, carousels) inherit a learned visual voice the way text generation inherits a learned tonal voice | None today — the type exists, the extractor doesn't, so the quality impact is currently zero despite the clear design intent |
| Asset-derived brand signal | Text-shaped knowledge extracted from non-text assets (logos, decks, brand guideline PDFs) | Lets brand voice accumulate from *artifacts the brand already has*, not only from content BrandOS itself generates — closes the cold-start gap faster for a new workspace that uploads existing brand materials on day one | Indirect but real — it feeds the same `learn()` pipeline as generated text, so a well-scored asset description can become a tone pattern or hook style just like a well-scored generated post |
| Explicit human satisfaction signal (orphaned) | What a human explicitly thought of a specific artifact | This is the *only* first-person, intentional, post-hoc quality signal in the entire system — every other signal is either inferred from text patterns or computed by a mechanical scorer | Currently zero, because nothing reads it — but its *potential* impact is the highest of any signal in this inventory, since it's the one source of evidence that isn't a proxy for quality, it's a direct human verdict |
| Human review decision | Whether a person explicitly approved or rejected a candidate signal | Adds a deliberate human gate to the automatic pipeline, when enabled | Currently low utilization (`require_review` defaults false) but architecturally the closest thing to ground-truth supervision the inferred system has |
| Generation quality score | A mechanical proxy for "was this good text" | The single most consequential piece of meta-knowledge in the whole system — it decides whether *anything* gets learned at all from a given artifact | Maximal indirect impact — a low score blocks all learning from that artifact regardless of how representative its style might otherwise have been |
| Repair status | Whether governance had to intervene before the artifact reached the user | Distinguishes "the model's natural output was good" from "the model's natural output needed fixing" — only the former should inform brand voice | High indirect impact — prevents learning from teaching the brand voice model what governance had to *correct away from* |
| Topic identity (hashed) | Whether a pattern generalizes across genuinely different topics, or is an artifact of one recurring topic | This is the mechanism that turns "we said this once about AI" into "we say this regardless of subject" — the actual definition of *style* vs. *topic* | Very high — without topic diversity weighting, the confidence model would reward a phrase used 10 times about one topic as highly as a phrase proven across 10 different topics, which is the wrong signal for *style* |
| Confidence | A 0–0.95 trust score per signal | Determines whether a pattern is strong enough to be worth injecting into a prompt at all | Direct — `StyleProjectionResolver` ranks and selects by weighted confidence; this is the actual gatekeeper of what reaches the LLM |
| Frequency | Raw recurrence count | One ingredient of confidence — recurrence alone is necessary but not sufficient | Indirect, via confidence |
| Topic diversity | Recurrence *across distinct contexts* | Weighted higher than raw frequency (45% vs 35%) — deliberately, because a brand's true style should hold regardless of subject | Indirect but the single most heavily-weighted ingredient of confidence |
| Style drift | Detecting that the brand's voice has evolved and an old pattern should yield to a newer one | Prevents the system from being "stuck" on an early voice as the brand matures | Protects against staleness — without it, a brand that intentionally shifts tone would keep getting old-voice signals injected indefinitely |
| Signal decay | Confidence erosion for unreinforced patterns | Keeps brand voice current rather than a permanent snapshot of however the brand wrote in its first month | Medium-high over long timescales; protects against a brand's voice model calcifying |
| Signal consolidation | Merging near-duplicate signals, enforcing a cap per type | Keeps the active signal set small, high-quality, and non-redundant, so the top-N selection at projection time is meaningful | Indirect but structurally necessary — without it, near-duplicate low-value entries would crowd out genuinely distinct, high-value ones at the selection stage |
| Workspace brand voice (declared) | A flat fallback set of tone/cadence/banned-phrases | The safety net for workspaces with no accumulated signal yet, and the explicit guardrail (banned phrases) layer | Medium — primarily a floor, not a ceiling; the explicit `banned_phrases` list does meaningful, direct quality work by suppressing AI-cliché language regardless of learned signal |
| Operating principle: topic-independence | A rule that persona/business identity must not bias topic selection | This is knowledge about *how to use* the other knowledge correctly — a brand whose persona says "Technology, Enterprise Leaders" must still be able to write well about an unrelated topic if asked | Directly protects against a known failure mode (topic drift) that was apparently serious enough to be fixed and explicitly documented in the source itself |
# Deliverable 3 — Evidence Model

BrandOS draws evidence from a narrower set of sources than the roadmap's own category list might suggest — confirmed by reading every call site that produces a `learn()` or `recordArtifactObservation()` invocation. The honest picture is: **two live evidence sources, one orphaned evidence source, and several sources that don't exist yet.**

| Evidence source | Live in BrandOS today? | What it actually contributes | Reliability | How IntelligenceOS should weight it |
|---|---|---|---|---|
| **Artifact generation (the artifact text itself)** | Yes — the dominant, near-exclusive source | Every Class A/B/C signal extraction in the entire inventory runs against `outputText`, the text BrandOS itself generated | Conditionally high: reliable *only* downstream of two gates (score ≥ 75, not repaired). Below those gates, contributes nothing. Above them, it's a real behavioral artifact, not a guess | Treat as the **primary, default-weight source** — but only after an equivalent quality gate. The gate is doing as much epistemic work as the extraction logic itself; without it, this source degrades to "whatever the model happened to output," which is circular as a brand-voice signal |
| **Uploaded assets (images, documents)** | Yes — via VLM/document analysis feeding the same pipeline | A synthesized text fragment (description + mood + recommendations, or description + topics) standing in for the actual asset content | Medium — mediated through an LLM's *description* of the asset, not the asset's literal content; the `confidence` field the VLM returns is itself stamped onto the score gate, so reliability is self-reported by the same model producing the content, with no independent check | Weight as **supplementary, not primary** — it's valuable precisely because it shortens cold-start (a brand's *existing* materials, not just what BrandOS generates going forward), but the self-graded confidence score should not be trusted at face value; consider a lower ceiling on confidence contribution than generation-derived signals get, since there's no analog to the "topic diversity across many generations" check that disciplines text-derived signals |
| **Prompt (the user's request text)** | Used for context-merging (e.g., `brandContextOverrides`), not for learning | The user's request can *override* declared persona fields for a single generation, but nothing about the prompt itself becomes a stored, persistent learning | N/A as an evidence source for learning (it's an evidence source for that one generation's *context*, a different question) | IntelligenceOS should explicitly decide whether prompts should ever become learning evidence (e.g., recurring phrasing the user keeps requesting) — BrandOS doesn't do this, so there's no behavioral precedent to inherit, only a clean opportunity (see Deliverable 7) |
| **Conversation (multi-turn dialogue)** | Does not exist — BrandOS has no multi-turn conversational generation surface | N/A | N/A | N/A as a BrandOS-derived requirement; IOS's own conversational surfaces (if any) would need their own evidence design with no BrandOS precedent to draw on |
| **Artifact edits / regeneration** | Does not exist — confirmed via the versions route's own doc comment: every generation is insert-only, no edit flow ships today | N/A | N/A | N/A as a BrandOS-derived requirement — but this is one of the clearest **evolution opportunities** (Deliverable 7), since edit deltas are usually the single highest-signal evidence source in comparable systems and BrandOS simply never built the surface to capture them |
| **User feedback (explicit rating + note)** | Collected, never read | A categorical satisfaction signal (`useful`/`generic`/`off_tone`/`excellent`/`needs_work`) plus free text, tied to a specific artifact | Potentially the **highest-reliability source in the entire system** — it's the only evidence that's a direct human verdict rather than a proxy (mechanical score) or an inference (text pattern extraction) — but currently contributes exactly zero, since nothing downstream reads the `feedback` table | IntelligenceOS should weight this as **the highest-trust source available**, specifically because it's scarce, intentional, and human — not because it will be the most *frequent* evidence (it won't; most artifacts will never receive explicit feedback) |
| **Human review (approve/reject a pending signal)** | Live, but gated behind `require_review`, which defaults to `false` | A binary supervisory decision over a specific *learned signal* (not over an artifact) | High when present — it's a direct human judgment about whether a *pattern* (not just one artifact) should be trusted | Treat as a **confidence override**, not just one more input to the weighted average — a human rejection should be capable of suppressing a signal that automatic scoring alone would have kept |
| **Workspace context** | Yes — feeds `brandContextOverrides`/`mergeBrandContext`, and is the join key for all stored signals | Scoping and a flat fallback voice description; not itself "evidence" so much as the *unit of account* that all other evidence is gathered within | N/A as an independent evidence source — it's the container, not the content | IOS should preserve workspace as the default scoping unit for inferred signals (per the already-validated Epic 1 plan) while still tracking user-level evidence separately, since BrandOS's own `personas` table shows the system already distinguishes "the workspace's declared voice" from "what an individual generation run actually produced" |
| **Persona (declared identity)** | Yes — explicit user input, read at generation time via `mergeBrandContext` | Tone, domain, audience, key themes — directly authored, not inferred | Maximally reliable *as a statement of intent*, but it's a snapshot the user wrote once (or rarely updates) — it doesn't tell you whether the brand's *actual output* matches what was declared | Treat as **a prior, not a likelihood** — a strong starting point that should be progressively superseded by accumulated inferred evidence as confidence in the inferred signals grows, exactly the precedence BrandOS's own design implies (declared voice is the fallback used when inferred signal is thin) |
| **External assets (e.g., competitor content, industry benchmarks)** | Does not exist anywhere in BrandOS | N/A | N/A | N/A as a BrandOS-derived requirement — flagged in Deliverable 7 as a genuine evolution opportunity with zero behavioral precedent to constrain the design |
| **Governance score (machine quality grade)** | Yes — the universal gate | Not itself a learning *category*, but the reliability multiplier on every other piece of evidence in System A | This is the most consequential "evidence about evidence" in the inventory — it's what makes the difference between "the brand says things this way" and "the model happened to output this once" | IntelligenceOS needs an equivalent gate regardless of which underlying scoring mechanism it uses — the *function* (filter low-quality output before it can become a learning input) matters more than BrandOS's particular cliché/buzzword heuristics, which IOS has no obligation to reproduce |
| **Repair status** | Yes — a second gate alongside score | Same function as the governance score: distinguishes natural model output from corrected output | High reliability as a binary fact, and cheap to check | Preserve the *principle* (don't learn from corrected output) even if IOS's correction/repair mechanism looks nothing like BrandOS's auto-fix pass |

**Cross-cutting evidence-reliability observation:** BrandOS's evidence model has an honest asymmetry worth naming directly — the system is very disciplined about *not learning from bad text* (two independent gates: score threshold, repair flag) but has built **no equivalent discipline for learning from text a human actually liked**, because the one human-verdict signal it collects is never consulted. A system that filters aggressively on a machine proxy for quality while ignoring the one human-provided quality signal it already has is mis-prioritizing its evidence sources, not just under-using one of them. This single observation should anchor IntelligenceOS's design more than any other finding in this evidence model.

# Deliverable 4 — Knowledge Characteristics

# Deliverable 6 — Native IntelligenceOS Design Recommendations

This is organized by *native IOS concept*, working backward from "what should this become" rather than forward from "what does BrandOS call it" — per the brief's explicit instruction. Each entry states the recommended representation, why it's a better fit than a literal port, and which BrandOS learning categories it absorbs.

### As a `Learning` (User Intelligence)

**Recommendation:** Tone pattern, hook style, format preference, narrative arc, executive cadence, argumentation style, evidence pattern, CTA intent — i.e., everything BrandOS calls "Class A/B" — should become `Learning` rows with `taxonomyCategory` values drawn from the existing 25-value enum (several map directly: `voice_tone`, `writing_style`, `communication_style` are exactly this), `stabilityClass: 'long_term'`, and `state` progressing through the existing `LearningState` machine (`ACTIVE → CONFIRMED → VALIDATED`, or `→ DECAYING → ARCHIVED` if unreinforced).

**Why this is better than a port, not just a rename:** BrandOS stores these as a flat, single `confidence` number per signal with no explicit state machine — a pattern is either "in the top-N by confidence" or it isn't, with no formal notion of "provisional," "corroborated three times," or "actively contradicted." `Learning`'s `state` field already gives IntelligenceOS exactly that vocabulary natively, and it costs nothing extra to use it, because the table already exists. The one place BrandOS is *more* sophisticated than a naive `Learning` row would be is its frequency/topic-diversity-weighted confidence formula — that logic should be preserved (see "As a `Hypothesis`" below for where the *promotion* path that produces a confident `Learning` should live), not the flat-confidence storage shape.

### As a `Hypothesis`, promoted to a `Learning` only after corroboration (User/Project Intelligence)

**Recommendation:** This is the single highest-value design upgrade available in this analysis. BrandOS's `recalculateConfidence()` — frequency × topic-diversity × source-quality, blended into one number — is functionally *trying to do* what `Hypothesis.requiredCorroborations`/`currentCorroborations`/`highQualityContradictions` already model explicitly and separately. Don't store a pattern directly as a confident `Learning` from the first observation. Store it as a `Hypothesis` first; let topic-diverse, quality-gated observations increment `currentCorroborations`; promote to `Learning` (via `promotedLearningId`) only once a corroboration threshold is met. **Domain position and market narrative are the clearest beneficiaries** — these are exactly the substantive-claim categories where BrandOS's own design (Deliverable 4) already recognized contradiction matters, but had no contradiction-tracking mechanism to use. `Hypothesis.highQualityContradictions` gives IntelligenceOS a native way to represent "the brand said X for a while, but three high-quality artifacts since have said not-X" — something BrandOS literally cannot represent today (Class C just accumulates).

**Why this is better:** It separates "we've seen this once" from "we believe this," with an explicit, inspectable threshold in between — rather than BrandOS's single rolling confidence number, which conflates "seen many times" with "believed strongly" in a way that's hard to audit or explain to a user asking "why does it think this about my brand?"

### As an `ArtifactPattern` with three-tier scope inheritance (Artifact Intelligence)

**Recommendation:** Title pattern, structural arc, value frame, and hook pattern (BrandOS's Class B — the domain-noun-abstracted structural templates) map directly to `ArtifactPattern.sections`/`narrativeModel`/`toneModel`, using the already-built `universal → archetype → user_calibrated` scope-resolution order in `ArtifactIntelligenceDomain.getPattern()`.

**Why this is better than a port:** BrandOS's `slotDomainNouns()` abstraction (replacing capitalized nouns with `[X]`/`[Y]` placeholders before storing a structural template) is a clever, hand-rolled way to approximate "this pattern, generalized across topics" — but it operates on a *flat list of signals per workspace*, with no notion of an artifact-type-universal baseline that every new user starts from. `ArtifactPattern`'s three-tier model gives every new user a sensible `universal` starting pattern on day one (BrandOS has no equivalent — a brand-new workspace has zero Class B signal and gets nothing structural until enough artifacts accumulate), lets an `archetype` tier generalize across users who share a content style without needing per-user history, and only specializes to `user_calibrated` once there's enough individual signal to justify it. This is a genuinely better cold-start story than BrandOS's design, achieved by using a domain IOS already built rather than copying BrandOS's flatter structure.

### As an `ArtifactExemplar`, promoted from a `FeedbackEvent` (Artifact Intelligence)

**Recommendation:** This is where BrandOS's orphaned `feedback` table evidence should land. `ArtifactExemplar.promotionReason: 'deployed' | 'explicitly_praised'` already models exactly the case where a human said "this was excellent" about a specific artifact — promote that artifact's `structuralSnapshot`/`voiceSnapshot`/`audienceSnapshot` into the exemplar table, and let future `ArtifactPattern` resolution preferentially draw from exemplars when constructing or reinforcing a `user_calibrated` pattern.

**Why this is better than what BrandOS does (or rather, fails to do):** BrandOS collects exactly this signal and discards it by never reading it back. IntelligenceOS doesn't even need to invent a new mechanism — `ArtifactIntelligenceDomain.recordFeedbackEvent()` and the `promoteExemplar()` stub already exist with the right shape; the gap is purely that `promoteExemplar()`'s trigger logic (when does a `'explicit_feedback'`-type `FeedbackEvent` with a positive rating actually fire a promotion?) needs to be designed and built, where BrandOS never even got as far as designing it.

### As a `KnowledgeAsset` (Knowledge Intelligence)

**Recommendation:** Named framework and domain theme (the purely archival half of BrandOS's Class C) map to `KnowledgeAsset.extractedFrameworks`/`extractedPatterns`, with `ownerType` set to `'user'` for an individual's own coined methodology or `'workspace'` for a shared one. This is also the right home for VLM/document-derived asset knowledge — `KnowledgeAsset.sourceFileRef` already models "this knowledge came from an uploaded file," which BrandOS's pipeline (asset → synthesized text fragment → generic signal row) flattens away entirely, losing the asset provenance.

**Why this is better:** `KnowledgeAsset` already has `version`/`isCurrent`, giving named-framework and domain-theme knowledge real version history for free — something Deliverable 4 flagged as a gap BrandOS never built for Class C. It also preserves *what the knowledge came from* (a specific uploaded file, vs. inferred from generated text) as a first-class field, which matters for explaining provenance to a user later ("we learned this from the brand guidelines PDF you uploaded" vs. "we inferred this from your last 12 posts" are different, useful things to be able to say, and BrandOS's flattened pipeline can't distinguish them after the fact).

### As a `Hypothesis` rather than a `KnowledgeAsset` specifically for domain position / market narrative

**Recommendation:** Distinguishing from the `KnowledgeAsset` recommendation above — domain position and market narrative are *evolvable claims*, not static reference material the way a named framework or an uploaded playbook is. Model them as `Hypothesis` rows (with eventual promotion to a `Learning` or, more likely, a dedicated read surface that exposes "current corroborated position" alongside "superseded positions") rather than `KnowledgeAsset` rows. The distinguishing test: would a user be surprised if this fact silently changed without an explicit edit? A named framework, no — it's authored. A domain position, yes — it's *inferred*, and a brand reversing position is a fact worth surfacing, not silently overwriting.

### As an `AudienceProfile` with `ownerType: 'generic'`, with an explicit upgrade path to `'named'`

**Recommendation:** BrandOS's flat `audience: string` field on `personas` should become a Phase-1 `AudienceProfile` with `ownerType: 'generic'`, `audienceType` drawn from the existing enum (`board`/`investor`/`engineering`/`customer`/`general`), and `expertiseLevel`/`communicationNorms`/`knownSensitivities` populated as evidence accumulates rather than left as an unstructured string. Critically, do **not** try to retrofit BrandOS's flat audience string into a named `Relationship` — BrandOS has no concept of a named recipient at all, so there's no migration path to design here, only a forward design decision.

**Why this is better:** A flat audience string can't express "this brand writes for engineers who are skeptical of marketing language" the way `expertiseLevel` + `knownSensitivities` can, and it gives IntelligenceOS a clean, already-typed activation path into full `Relationship` modeling later (per the existing Phase 2 trigger: "≥3 external artifacts with named recipients") without needing a redesign when that day comes.

### As a `Project.goals` / `Project.constraints` entry — net-new capability, no BrandOS precedent

**Recommendation:** Since BrandOS has no goal or constraint model at all (confirmed: the planner agent does tactical idea generation with no strategic-intent representation), this is pure greenfield design within IOS's existing `Project` entity, which already has the fields. Recommend modeling a goal as a structured object (not a free string) with at minimum a description, a target artifact type or campaign association, and a status (active/achieved/abandoned) — giving `BlueprintBuilder` something concrete to calibrate `depthSpec`/`narrativeFrame` against beyond "what topic was this artifact asked to be about."

### As a `VocabularyDirectives`-feeding signal, not folded into `Learning`

**Recommendation:** BrandOS doesn't really have a standalone "vocabulary" learning category — `banned_phrases` is the closest thing, and it's a flat, declared list with no inference behind it at all. IntelligenceOS's `VocabularyDirectives` (already "fully specified verbatim" per its own doc comment — the most stable, trustworthy field in the blueprint type) deserves a real evidence pipeline BrandOS never built: `preferredTerms`, `domainJargon`, and `proprietaryTerms` should be inferable from repeated word-choice patterns in well-scored artifacts (a genuinely new extraction category — see Deliverable 7), while `forbiddenTerms` should remain a directly-declared list, mirroring BrandOS's `banned_phrases` precedent for the one piece of vocabulary knowledge it does handle well (a hard, user-declared floor, never inferred, never overridden by accumulated signal).

### As `WorkspaceIntelligenceDomain` data, decoupled from `UserIntelligenceDomain`'s declared identity

**Recommendation:** BrandOS's `personas` table conflates two genuinely different things under one row: "what does this individual say their voice sounds like" and "what does this entire workspace require regardless of who's writing." IntelligenceOS's domain split already separates these correctly in principle (`UserIntelligenceDomain` vs. `WorkspaceIntelligenceDomain`) — the design recommendation is to make sure declared identity, when it's first modeled, is explicitly *attributed* to one domain or the other rather than defaulting to "whatever's on the persona row," the way BrandOS effectively does. A practical test: if a team has five people each with their own user-level tone but a shared "never say X, always mention compliance disclaimer Y" — the first half is `UserIntelligenceDomain`, the second is `WorkspaceIntelligenceDomain`, and BrandOS's single-table design has no way to express that split even though its own architecture (separate `brandContext` merge logic) gestures at wanting one.

### As an `IdentityVersionRecord`-style mechanism for visual identity — finally given a real producer

**Recommendation:** BrandOS already designed (`identity-types.ts`) exactly the right shape for this — `IdentitySignal`, `MergeStrategy`, `IdentityVersionRecord` — and then never built the extractor that would feed it. IntelligenceOS shouldn't re-derive this design from scratch; it should treat BrandOS's dormant type system as a *correct but unimplemented spec* and build the missing half: a visual-asset analyzer that produces structured `IdentitySignal`s for color/typography/layout/mood dimensions (not just the two fields — palette and typography personality — that BrandOS's narrower VLM merge actually captures today), feeding into a `Learning`-equivalent representation with the same decay/confidence machinery as text-voice learning, rather than the flat "union the palette array, cap at 6" merge BrandOS settled for.

### As cross-cutting middleware, not a domain — the quality/repair gate

**Recommendation:** Preserve the *principle* of BrandOS's two-gate evidence filter (quality score, repair status) as an invariant enforced wherever `IntelligenceOS.recordFeedbackEvent()` or any future signal-extraction pipeline ingests a new observation — but don't let it ride on BrandOS's specific mechanical scorer. A better-designed version should let multiple evidence-reliability signals combine (mechanical quality score *and* explicit human feedback *and* deployment/edit behavior), rather than BrandOS's single rule-based gate being the only thing standing between "model output" and "stored brand truth."

### As an explicit invariant in blueprint assembly — topic-independence

**Recommendation:** Preserve BrandOS's `TOPIC-DRIFT-FIX-002` principle directly in `NarrativePlanner`/`StructurePlanner`'s design: persona-level or project-level *identity* signals (industry, audience, declared themes) must never be allowed to override or bias the *topic* the user explicitly asked for in a given request. This is worth stating as an explicit design constraint precisely because BrandOS had to discover and fix this as a real regression — IntelligenceOS gets to inherit the lesson without inheriting the bug.
# Deliverable 7 — Capability Evolution Opportunities

Organized by opportunity, each tied back to the specific BrandOS gap or dormant design that motivates it.

### 1. Close the orphaned-feedback loop

BrandOS collects explicit human satisfaction signal (`useful`/`generic`/`off_tone`/`excellent`/`needs_work` + note) and never reads it back. This is not a hypothetical opportunity — it's a fully-specified evidence type sitting unused, with a native IOS home (`ArtifactExemplar`, `FeedbackEvent.eventType: 'explicit_feedback'`) already built and waiting. This is the single highest-confidence, lowest-design-risk evolution opportunity in this entire analysis, because it requires no new schema and no new evidence-collection surface — only a consumption path that doesn't exist yet in either system.

### 2. Build the edit-diff evidence source BrandOS never built a UI for

BrandOS has no regenerate/edit flow at all (confirmed via the versions route's own doc comment). `FeedbackEvent.editDiff`/`SignalSourceType: 'edit_diff'` are already typed in IOS with no producer. When BrandOS or any consumer eventually ships an edit surface, the diff between what was generated and what the user kept should become a *direct* learning signal — frequently the highest-information evidence type in systems with this shape, because it shows exactly where the model's instinct diverged from what the human actually wanted, word for word, rather than relying on a coarse post-hoc rating.

### 3. Multi-signal evidence reliability, not a single mechanical gate

BrandOS gates all learning on one rule-based score. IntelligenceOS can do better by combining multiple independent reliability signals — mechanical quality score, explicit feedback (when present), deployment/usage behavior (was the artifact actually published, vs. generated and discarded) — into a richer trust function, rather than a single pass/fail threshold. This also opens a path to *partial* trust: an artifact that scores well but was never used shouldn't be weighted the same as one that scored well and was deployed.

### 4. Real contradiction handling for substantive knowledge

BrandOS's Class C (domain position, market narrative) accumulates without ever resolving conflicting claims. `Hypothesis.highQualityContradictions` already gives IntelligenceOS the scaffolding to detect and surface "the brand used to claim X, recent evidence suggests not-X" as an explicit, reviewable event — rather than either silently overwriting (losing history) or silently accumulating both (never resolving the tension), which is what BrandOS effectively does today.

### 5. Cold-start quality via universal/archetype pattern tiers

BrandOS gives a brand-new workspace nothing structural until enough of its own artifacts accumulate. `ArtifactPattern`'s `universal → archetype → user_calibrated` hierarchy (already built in IOS) means a new user gets a sensible baseline on day one and personalizes from there, rather than starting from zero the way BrandOS does. This is a direct, immediate quality improvement for new users with no BrandOS analog to even reference.

### 6. Cross-workspace / cross-user learning at the archetype tier

Once `ArtifactPattern.archetypeType` is populated for real, IntelligenceOS could let users who share an archetype (e.g., two different B2B SaaS founders) benefit from pattern signal pooled across users at that tier — something BrandOS's strictly per-workspace signal model structurally cannot do (every `brand_memory_entries` row is workspace-scoped with no aggregation across workspaces). This needs careful design around what's safe to pool (structural/stylistic patterns, almost certainly yes; anything touching Class-C-equivalent substantive content, almost certainly no) but the domain separation IOS already has (Artifact Intelligence's structural patterns vs. Knowledge Intelligence's substantive content) makes this a much safer thing to attempt than it would be against BrandOS's flatter signal model.

### 7. A real visual-identity extractor behind the already-designed types

`ISemanticIdentity`/`IVisualIdentity`/`IdentityVersionRecord` are fully specified and never implemented. Building the missing extractor (beyond BrandOS's narrow palette+typography VLM merge) would let IntelligenceOS support visual-artifact generation (decks, carousels) with a learned visual voice the way text generation already gets a learned tonal voice — a capability gap BrandOS clearly intended to close and never did.

### 8. Vocabulary modeling beyond a banned-phrase blocklist

BrandOS's only real vocabulary signal is a declared blocklist. A genuine extraction pipeline feeding `VocabularyDirectives.preferredTerms`/`domainJargon`/`proprietaryTerms` from repeated word-choice patterns in well-scored artifacts would give IntelligenceOS positive vocabulary guidance ("this brand says 'partners' not 'clients'"), not just negative guidance ("never say synergy") — a capability category BrandOS never attempted.

### 9. Named relationship and calibrated-audience modeling

BrandOS's audience field is a flat string with zero named-recipient concept. IOS's `RelationshipIntelligenceDomain`/`AudienceProfile` (`ownerType: 'named'`) already has activation triggers defined for exactly this. There's no BrandOS precedent to port from, but that also means no legacy constraint shaping the design — this can be built well from first principles when the activation triggers fire.

### 10. Goals and constraints as first-class, structured project knowledge

BrandOS has no representation of strategic intent at all. `Project.goals`/`Project.constraints` already exist as typed fields with no current writer. Designing a real structured-goal model (description, target association, status) would let `BlueprintBuilder` calibrate depth and narrative framing against actual strategic intent, not just topic and audience — something BrandOS's purely tactical planner agent has no ability to do.

### 11. Provenance-aware knowledge, not flattened signal

BrandOS's asset-analysis pipeline synthesizes a generic text fragment from an uploaded asset and feeds it into the same anonymous signal pipeline as generated text, losing the fact that this particular piece of knowledge came from a specific uploaded file. `KnowledgeAsset.sourceFileRef` already preserves this. Surfacing provenance to the end user ("we learned this from your uploaded brand guidelines" vs. "we inferred this from your last dozen posts") is a trust- and explainability-building capability with no BrandOS analog.

### 12. Temporal/versioned substantive knowledge with explicit supersession

BrandOS's Class C never decays and never versions — a domain position from two years ago and one from last week sit at equal standing. A `Hypothesis`-to-`Learning` promotion path with explicit `promotedLearningId` linkage, plus a "superseded by" relationship for substantive claims, would let IntelligenceOS answer "what does this brand currently believe about X, and has that changed" — a question BrandOS's design cannot answer today even in principle.

### 13. Confidence as corroboration-and-contradiction, not a blended formula

BrandOS blends frequency, topic diversity, and source quality into one opaque number. `Hypothesis`'s explicit `requiredCorroborations`/`currentCorroborations`/`highQualityContradictions` fields make the *reasoning* behind a confidence level inspectable and explainable — a meaningfully better foundation for any future "why does the system think this about my brand" UI than a single blended float.

# Deliverable 8 — Epic 1 Recommendations

The mandate here is narrow and explicit: identify only where this semantic analysis changes *what an Epic 1 implementation task should do*, not where it merely adds color. Three of Epic 1's five features change in a real, specific way. Two do not change at all — they're confirmed correct as scoped. One correction to the Implementation Guide's own prior framing of E1-4 is folded in here since it directly affects how that task should now be approached.

### E1-1 (Human Learning Review API) — strengthened, scope widened by one finding

**Learning objective:** Human review exists in BrandOS as a supervisory override on an *automatically inferred* signal (approve/reject a candidate Class A/B/C entry). This semantic analysis surfaces a second, more valuable kind of human signal BrandOS collects but never connects to review at all: explicit post-hoc satisfaction feedback (`useful`/`excellent`/`off_tone`/etc.).

**Why this matters for artifact quality:** A review API that only ever processes machine-proposed candidate signals captures supervisory correction, but misses the higher-trust evidence category identified in Deliverable 3 — direct human verdicts on finished output.

**Recommended IOS design:** `reviewLearning()` as already scoped in the Implementation Guide (approve/reject a `FLAGGED` learning) is correct and should ship unchanged. **Add** a second, related task to Epic 1: ensure `IntelligenceOS.recordFeedbackEvent()`'s existing `FeedbackEvent.eventType` enum is exercised for `'explicit_feedback'` events with a path that can eventually feed `ArtifactExemplar` promotion (Deliverable 6/7, item 1) — not full exemplar-promotion logic itself (that's rightly a later-sprint capability per `ArtifactIntelligenceDomain`'s own stub comments), but confirming the `recordFeedbackEvent()` write path correctly persists an explicit-feedback event today, so a Sprint-3-style promotion job has real data to work from once it's built. This is a small, additive verification/wiring task, not a redesign, and it directly closes the gap Deliverable 3 identifies as the single most consequential evidence-prioritization mismatch in BrandOS's design.

### E1-2 (Workspace-Scoped Brand Voice) — strengthened, one modeling correction

**Learning objective:** "Workspace brand voice" in BrandOS's actual behavior is a flat, declared fallback (`tone`/`cadence`/`banned_phrases`), not an inferred, evolving body of knowledge the way user-level tone patterns are. The Implementation Guide's E1-2 design (treat it as `Learning` rows scoped by `workspace_id`, with the same decay/confidence machinery as user-level learnings) inherits the *user-level* learning shape for a category that, per Deliverable 5/6's domain mapping, is better split into two genuinely different things: a declared floor (workspace-level constraints, closer to `WorkspaceIntelligenceDomain`'s existing compliance-constraint reading) and an inferred, evolving layer (only this part should actually use the `Learning`-with-decay machinery).

**Why this matters for artifact quality:** Treating a hard declared constraint (e.g., a workspace-wide compliance disclaimer requirement) as if it were a decaying, confidence-weighted inferred pattern risks letting it silently weaken over time the way an unreinforced tone pattern correctly should — but a compliance requirement should never decay just because it wasn't "reinforced" recently.

**Recommended IOS design change:** Within E1-2's existing task structure (no new Feature, no schedule change), split `getWorkspaceLearnings()`/`upsertWorkspaceLearning()`'s scope: declared, never-decaying workspace constraints should route through `WorkspaceIntelligenceDomain.getContext()`'s existing `complianceConstraints` mechanism (already built, already correctly non-decaying) rather than being modeled as workspace-scoped `Learning` rows at all; only genuinely inferred, evolvable workspace-level *style* patterns (if any turn out to exist — e.g., "this workspace's team consistently writes shorter copy than any individual member's personal baseline") should use the `Learning`-with-decay path E1-2 already designs. This doesn't change E1-2's task list or estimate meaningfully — it changes which of the two new methods a given piece of workspace knowledge should call, and should be reflected as an explicit decision point inside E1-2-T2's implementation, not a new task.

### E1-3 (Brand Summary Query API) — confirmed correct, no change

**Learning objective:** `getBrandSummary()` is a read-side aggregation of already-modeled `Learning`/`Archetype` state for human-facing display. Nothing in this semantic analysis identifies a different *learning objective* behind this capability — it is purely a presentation aggregation over knowledge other Epic 1 features already produce.

**Why no change:** The richer domain mapping developed in this analysis (Deliverables 5/6) doesn't change what should be *summarized*, only confirms that `activeLearningsCount`/`topTaxonomyCategories` are reading from the right underlying source (`Learning.state`/`Learning.taxonomyCategory`, both already correctly scoped). E1-3 ships as designed.

### E1-4 (VLM Visual Intelligence Bridge) — strengthened, and one factual correction to the Implementation Guide

**Correction first, since it changes the premise of this task:** The Implementation Guide characterized this bridge as not-yet-built on the BrandOS side ("E1-4-T3... tracked, not built in Epic 1... the real `bus.emit()` call cannot be added to BrandOS until an `IntelligenceOS`/`IntelligenceOSProvider` instance is reachable"). Direct source inspection for this analysis found that **BrandOS already feeds VLM-derived asset analysis into its brand-memory learning pipeline today**, via `recordBrandMemoryObservation()` called directly from `apps/web/app/api/assets/[id]/analyze/route.ts` — no new infrastructure was needed on the BrandOS side for *that* bridge, because it reuses the same proxy function path already wired for text-artifact observations. This doesn't invalidate E1-4's IOS-side task list (the IOS-side event type and handler still need to be built exactly as scoped), but it means the eventual end-to-end wiring (previously deferred to E3-M1) is mechanically simpler than described — it's adding a second call alongside an existing, working pattern, not building a bridge from nothing.

**Learning objective, refined by this analysis:** Deliverable 1/2's reading of the actual VLM prompt (`IMAGE_ANALYSIS_PROMPT`) shows BrandOS is trying to learn `description`/`colors`/`typography`/`mood`/`recommendations` from an image — and Deliverable 1 separately surfaces that BrandOS also does a **second, completely independent visual-learning operation** on the same VLM result: merging palette and typography directly into `personas.visual_style`, entirely outside the brand-memory signal pipeline.

**Why this matters for artifact quality:** The Implementation Guide's E1-4 design (one new IOS event type, one new handler, folding VLM output into the existing taxonomy-tagged `Learning` pipeline as a `personal_brand_signal`-categorized entry) captures only the *first* of these two BrandOS behaviors. It silently drops the second — the direct, separate visual-identity merge — which is exactly the dormant `ISemanticIdentity`/`IVisualIdentity` capability gap identified in Deliverables 5–7 (evolution opportunity #7).

**Recommended IOS design change:** E1-4's IOS-side task list (new event type, new handler in `KnowledgeProcessor`) should ship as already scoped — that correctly captures the text-pipeline half. **Add** one Epic-1-scoped design note (not a new sprint-sized task, but something that should be decided before E1-4's handler is finalized): when designing `BrandAssetAnalyzedPayload`'s shape, include the structured `colors`/`typography` fields distinctly from the free-text `description`/`mood`/`recommendations` fields, rather than collapsing everything into one signal-text blob the way BrandOS's `recordBrandMemoryObservation()` call does (it joins description + recommendations + topics + mood into a single string before storage). This keeps the door open for the visual fields to eventually feed a proper visual-identity `Learning`/`IdentitySignal`-equivalent (evolution opportunity #7) instead of being permanently flattened into a generic text signal the way BrandOS's design forces them to be today. This is a one-line shape decision inside E1-4-T1's existing payload-design work, not new scope.

### E1-5 (Classification Backward-Compatibility Mapping) — confirmed correct in direction, calibration source corrected

**Learning objective:** This task exists purely to bridge IOS's richer model down to BrandOS's real three-value classification for the transition period — it has no independent learning objective of its own; it's a compatibility shim, and this analysis doesn't change that framing.

**One factual correction carried forward from this deeper read:** The Implementation Guide's E1-5-T2 (threshold calibration) cited a file, `BrandSignalClassifier.ts`, as the ground-truth calibration source. E1-5-T2's calibration work should be re-pointed at these two real artifacts. This doesn't change the task's complexity or sequencing, only its citation — but it should be corrected before someone goes looking for a file that isn't there.
